# v1.6 洞察与执行摘要

## 1. 核心洞察

这个仓库表面是“长截图切割器”，实际架构主题是“如何把一个浏览器端图片处理算法产品化”。它的核心价值链不是单个切割函数，而是：

上传与校验 → Worker 图片处理 → 内容感知切割与回退 → 状态层稳定排序和资源管理 → 预览选择 → PDF/ZIP 导出 → SEO/i18n/导航和部署把工具包装成可上线产品。

这种设计适合中等规模前端工具。它没有引入后端、微前端或复杂状态库，而是使用 React 组件、custom hooks、Worker、工具函数和目录边界维持复杂度。

## 2. 设计亮点

- 切割算法与 Worker I/O 分离，`splitAnalyzer` 可独立测试，降低算法迭代成本。
- 切片按 index 写入状态，专门防御异步 `img.onload` 乱序，是很实际的浏览器工程细节。
- 导出 UI 与 PDF/ZIP 工具解耦，主 App 统一接线，利于替换导出实现。
- 导航不仅靠按钮禁用，还在路径层做状态恢复，能处理刷新或手动 hash 输入。
- 扁平单仓库对当前项目规模是合理选择，文档 ADR 与实际目录大体一致。

## 3. 主要问题

- Worker 协议缺少 task id/cancel，连续上传、旧任务回写和取消处理的边界不够强。
- 大图仍一次性 `getImageData`，Worker 只解决主线程阻塞，没有解决内存峰值。
- `ExportControls` 的高级选项没有在 App 主路径完整传给 exporter，UI 能力和导出能力存在断层。
- SEO/i18n 语言来源不统一，用户切换英文后 head 元数据仍固定中文。
- 构建/部署辅助脚本有 monorepo 遗留假设，和当前“扁平单仓库”存在漂移。
- 部署 workflow 把部分质量检查失败降级为日志，发布质量门偏软。

## 4. 如果重新设计

优先不换框架，而是收敛边界：

1. 给 Worker 消息增加 `taskId` 与取消语义，主线程只接收当前任务的结果。
2. 把 `splitAnalyzer` 的全图像素读取改成分块统计，降低大图内存峰值。
3. 统一导出配置流：`ExportControls` 的 quality/pdfOptions/zipOptions 应进入 exporter，或删掉未生效 UI。
4. 将 `PageType`、导航步骤、SEO 页面配置、路由路径收敛为单一页面定义表。
5. 清理或接入 `tools/build-scripts` 中面向 `packages/*` 的脚本，避免基础设施误导。
6. 部署 workflow 至少让 type-check/test 失败阻断部署。

## 5. Budget Profile 执行摘要

预算目标：标准分析。

实际执行：

- Evidence 深度：覆盖 4 个核心模块，核心判断均要求源码锚点。
- 风险抽样强度：每个核心模块至少一条风险路径，重要边界追加抽样。
- Subagent 数量：4。
- 外部调研：只用本地 README/docs/source，不联网。
- 报告边界：完整标准报告；未展开快速/深度模式的额外运行。

因此，本次能够验证 v1.1、v1.2、v1.3、v1.4、v1.5、v1.7 的关键验收项；v1.6 的“预算档已记录并影响计划”通过，但“三模式分别运行并对比”未满足，只能判为部分通过。
