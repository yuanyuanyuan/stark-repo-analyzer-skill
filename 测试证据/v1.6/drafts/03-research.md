## 项目解决的核心问题

项目 README 将其定位为“长截图分割器”：一个基于 React + TypeScript + Vite 的长截图分割工具，支持按指定高度分割长截图，并导出为 PDF 或 ZIP：`README.md:1-3`。

具体场景：

1. 内容运营或设计人员拿到超长页面截图，需要切成适合分享、归档或移动端浏览的若干片段。
2. 用户希望在浏览器内完成处理，不把图片上传到后端服务。
3. 用户还需要导出 PDF/ZIP，而不仅仅是单张预览图。

README 的功能清单显示项目覆盖多图片格式、智能分割、PDF/ZIP 导出、响应式布局、多语言和可配置分割参数：`README.md:5-14`。

## 项目自己的架构定位

项目文档自述采用“扁平化单仓库架构”：`README.md:16-23`。架构文档 ADR 进一步说明选择该方案是因为项目规模适中、开发效率优先、维护成本较低和团队协作更简单：`docs/ARCHITECTURE.md:11-35`。

这提供了本次分析的核心假设：项目不是在追求企业级微前端或插件生态，而是在一个中等规模工具应用里，用单仓库和明确目录分层换取开发效率。

## 同类项目与替代路线

本次未做外部联网竞品调研。基于项目定位，只能提出候选替代路线，不能写成外部事实：

| 替代路线 | 可能优势 | 可能代价 |
|---|---|---|
| 纯本地桌面工具 | 文件系统能力强，适合大图批处理 | 安装成本高，跨平台分发成本高 |
| 后端图片处理服务 | 可承载更重的图片处理任务 | 上传隐私风险、服务成本、网络依赖 |
| 浏览器 canvas 小工具 | 零安装、隐私友好 | 受浏览器内存、Worker、Blob 限制影响 |
| 设计工具/截图工具内置裁切 | 生态内完成 | 通常不专注长截图批量切片和 PDF/ZIP 导出 |

## 为什么需要单独做这个项目

从 README 与架构文档看，项目把“长截图切割 + 导出 + 多语言 + SEO + 移动端”打包成完整 Web 应用，而不是只写一个算法函数。它的价值不只在分割算法，也在把上传、处理、预览、导出、响应式和部署组合成一个可上线工具。

该判断有文档证据支撑：功能清单见 `README.md:5-14`，分层架构图见 `docs/ARCHITECTURE.md:38-111`，数据流见 `docs/ARCHITECTURE.md:197-223`。
