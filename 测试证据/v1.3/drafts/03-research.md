# 阶段 3：外部调研与项目文档研读

## 项目解决的核心问题

Long_screenshot_splitting_tool 解决的是「长截图交付困难」问题：长图直接插入文档、聊天或打印时容易被压缩、阅读成本高，用户需要把它拆成多个更可控的片段，再导出为 PDF 或 ZIP。

源码与 README 共同说明它是一个纯前端工具：React + TypeScript + Vite，支持 PNG/JPG/JPEG/WebP/GIF，按指定高度分割长截图，并导出 PDF 或 ZIP。外部 GitHub 页面也将其描述为 client-side-only 的长截图智能分割工具。

## 竞品与同类项目定位

| 类型 | 代表 | 路线 | 与本项目差异 |
|------|------|------|--------------|
| 固定高度在线切图 | MeTool Long Image Splitter | 固定高度 + 自定义切线 | 更强调手动精确切线，本项目更强调浏览器内自动处理与导出流水线 |
| 截图扩展 | GoFullPage / FireShot | 先捕获整页，再导出图片/PDF | 解决「如何得到长截图」，本项目解决「已有长截图如何拆分和交付」 |
| 内容感知分割研究/项目 | Web-page-Screenshot-Segmentation | 低变化区域寻找切线 | 与本项目 splitAnalyzer 思路接近，但本项目还包含完整 Web UI、Worker、预览、导出 |
| 本地桌面工具 | XnView / ShareX 需求讨论 | 手动裁剪或打印分页 | 更偏桌面工作流，本项目强调零安装、浏览器端完成 |

## 为什么需要单独做这个项目

单独价值不在「切图算法」本身，而在把上传、后台分割、预览选择、PDF/ZIP 导出组合为一个无后端依赖的任务闭环。对普通用户来说，单个工具比「截图扩展 + 图片编辑器 + PDF 工具」的组合链路更短。

## 项目背后的组织动机

从 README、docs 与源码看，这是一个工具型个人/小团队项目。它同时投入 SEO、移动端优化、构建部署和 shared-components，说明目标不只是 demo，而是希望成为可公开访问、可传播、可长期维护的在线工具。

## 文档可信度观察

`docs/ARCHITECTURE.md` 与真实源码存在偏差：文档里描述了一些更宏大的架构能力，但当前源码主链路仍是单页 React 工具。v1.2 报告以源码锚点为准，文档只作为背景材料。
