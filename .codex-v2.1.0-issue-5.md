## What to build

建立 DeepWiki shadow mode 的 A/B benchmark 和 v2.1.0 发布门，用相同 commit、模型、模式、预算和并行度比较基线流程与 DeepWiki 增强流程。

该切片必须把效率和质量放在同一决策面板中：既记录 Evidence Plan 时间、源码读取量和 token，也盲审模块召回、关系与风险发现、无效锚点、Unsupported Claims 和错误确定性结论。完成后形成可复核的启用决策，并把最终默认策略、用户控制和已知限制写入 v2.1.0 发布文档。

## Acceptance criteria

- [ ] benchmark 至少覆盖小、中、大公共仓库，并包含文档丰富/稀缺、单体/monorepo 和不同扩展机制。
- [ ] 每个样本固定仓库 commit、模型、模式、总预算和并行度，基线与 shadow 使用可比较条件。
- [ ] 记录 Evidence Plan 墙钟时间、DeepWiki 调用成本、近似 token、本地读取文件和 span、候选验证率和首次 gate 结果。
- [ ] 质量评审记录核心模块 recall、跨模块关系、风险路径、无效锚点、Unsupported Claims 和错误确定性结论。
- [ ] 单独覆盖未索引、超时、服务失败和错误响应的无损降级测试。
- [ ] 长期启用必须满足父规格的效率或质量收益门槛，且错误结论、无效锚点和 gate 失败率不得恶化。
- [ ] 未达到收益门槛时维持手动/shadow 选项，不默认影响正式 Evidence Plan。
- [ ] 形成可复核 benchmark 报告，并在 v2.1.0 用户文档中说明默认策略、调用预算、隐私边界和已知限制。
