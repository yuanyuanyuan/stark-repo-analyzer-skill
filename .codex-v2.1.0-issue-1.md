## What to build

交付 DeepWiki Scout 的首个 shadow-mode 纵向闭环：对已经完成本地预检和确定性扫描的公共 GitHub checkout 做资格判断，通过可替换 provider 调用 Python CLI 获取 wiki structure，并把成功、未索引、超时和服务错误统一写成版本化研究工件。

该切片只记录候选和运行指标，不改变 Evidence Plan、coverage、Evidence Matrix 或最终报告。完成后，维护者可以在一个真实或 fixture 仓库上演示“可用时产出 external hypothesis，不可用时无损回到基线”的完整行为。

## Acceptance criteria

- [ ] 一个符合条件的公共 GitHub checkout 可以通过 Repo Analyzer 的外部命令入口生成标准化 DeepWiki 研究工件。
- [ ] 工件记录 schema/provider 版本、仓库标识、本地 commit、分析模式、获取时间、请求、原始响应、延迟、响应规模、索引状态、验证状态和错误原因。
- [ ] DeepWiki 输出默认标记为 `external_hypothesis` 和未验证状态，不改变任何正式证据工件。
- [ ] 未索引、超时、限流、协议失败和 HTTP 成功但正文失败均被识别为明确状态，并触发后续调用 circuit breaker。
- [ ] 本地无公共 GitHub 远端、私有仓库、脏工作区或远端分支不匹配时默认跳过，并记录可审计原因。
- [ ] 默认路径只调用 structure，不调用完整 wiki contents。
- [ ] 默认测试使用 fake provider，覆盖成功和全部降级路径，不依赖实时网络。
- [ ] DeepWiki 失败不影响现有 doctor、scan、summarize、units 和 gate 的行为。
