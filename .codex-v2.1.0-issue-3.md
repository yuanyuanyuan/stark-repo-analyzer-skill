## What to build

在 standard 模式交付 P0 Evidence Plan、DeepWiki gap challenge、本地验证和 P1 Evidence Plan 的完整闭环。

主 Agent 必须先在未接触 DeepWiki 内容的情况下保存独立 P0；随后使用一次 structure 和最多两个围绕核心数据流、模块边界或风险路径的定向问题，将结果分类为 baseline-only、DeepWiki-only、agreement 和 conflict。只有在当前 checkout 上完成源码验证的新增候选才能进入 P1 的正式范围，其余内容保留为开放问题或被拒绝线索。

## Acceptance criteria

- [ ] standard 模式在 DeepWiki 内容进入上下文前生成并保留独立 P0 Evidence Plan。
- [ ] gap challenge 最多执行一次 structure 和两个定向问题，并遵守总时间和上下文预算。
- [ ] P0 与 DeepWiki 的比较结果明确区分 baseline-only、DeepWiki-only、agreement 和 conflict。
- [ ] agreement 不被当作独立双源验证；DeepWiki-only 和 conflict 默认保持未验证状态。
- [ ] 进入 P1 的 DeepWiki 增量候选全部具有当前 checkout 的本地源码锚点和验证判断。
- [ ] 无法验证的候选进入开放问题、限制或被拒绝线索，不进入覆盖率和确定性叙事。
- [ ] DeepWiki 未索引或失败时，P0 直接成为正式 Evidence Plan，后续流程无损继续。
- [ ] 研究工件记录新增候选数、验证成功数、无效/陈旧候选数和累计调用成本。
- [ ] 端到端 fixture 测试覆盖有效增量、陈旧路径、冲突结论和完全降级四种流程。
