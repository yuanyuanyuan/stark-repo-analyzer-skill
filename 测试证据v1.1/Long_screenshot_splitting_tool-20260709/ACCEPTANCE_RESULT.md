## 验收项

### 1. 无效阅读成本下降 ✅

v1.1 实际阅读 ~5,800 行 vs v1.0 估算 ~18,800+ 行，降幅 ~69%。v1.1 未为凑覆盖率铺读低价值源码（SEO 组件 12.6K 行仅抽样读了 2 个关键文件，shared-components 仅读了接口定义）。大文件（App.tsx 745 行）定位关键段落精准读，未机械分段铺满。

### 2. 核心结论都有源码锚点 ✅

4 个核心模块草稿共附 59 条源码锚点（split-pipeline: 16、state-management: 16、routing: 18、export: 9），每个关键判断（角色、流程、设计决策、权衡）均带 `文件:行号`。抽查 3 个锚点全部与源码一致：
- `split.worker.js:125-129`：catch → splitPoints=[] 等分回退 ✅
- `useAppState.ts:51`：按 index 写入非 push ✅
- `App.tsx:124-135`：useRef + useEffect 监听 0→>0 自动跳转 ✅

### 3. 核心模块未被跳过 ✅

## 关键发现

v1.1 锚点机制在这个仓库上表现尤为突出——核心业务逻辑（切割流水线）仅 ~3000 行，但 SEO/i18n/mobile 噪音占了 70%+ 代码量。v1.0 覆盖率门控会强迫 agent 读大量 SEO 组件凑行数，而这些组件与「长截图如何被分割」这一核心问题毫无关系。v1.1 让 agent 聚焦核心流水线，用源码锚点保证结论可信度，同时用「次要模块抽样」处理周边代码——既不遗漏核心模块，也不浪费 token 读 SEO Manager 的 365 行。
