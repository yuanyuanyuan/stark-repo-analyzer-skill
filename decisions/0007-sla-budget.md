# ADR-0007 激进 SLA 预算：30 分钟 / 500K tokens / 3 次失败回退

## Context

整个 repo-analyzer skill 跑一次「完整尽调一家中型仓库（~10 万行代码）」得设个预算上限，否则每次跑心里没底、debug 难、SLA 无法对外承诺。

预算覆盖三件事：

1. **时间预算**：完整跑完允许多久？影响用户等待体验。
2. **token 预算**：调用 LLM API 花多少钱？影响单次成本。
3. **失败预算**：允许重试几次？超过就放弃。

候选方案：

| 方案 | 时间 | token | 失败回退 | 适合 |
|------|------|-------|---------|------|
| 激进 | 30 分钟 | 500K（≈ $5） | 3 次 | 试误 / 调试 |
| 平衡 | 90 分钟 | 1.5M（≈ $15） | 5 次 | 生产 |
| 保守 | 180 分钟 | 3M（≈ $30） | 10 次 | 政企 |

## Decision

**采用激进方案**：30 分钟 / 500K tokens / 3 次失败回退。

具体执行规则：

1. **时间预算 30 分钟**：
   - 单次完整跑（Phase-2a + 12 维精切 + 阶段三~八）硬上限 30 分钟
   - 计时器在阶段一（旧方案的「阶段一宏观认知」）被砍后改为从 **Phase-2a 启动** 开始计
   - 计时精度 1 秒，跨过 30 分钟立即 `yield` 终止

2. **token 预算 500K**：
   - 单次完整跑所有 LLM API 调用 token 加和上限 500K（含 input + output）
   - 阶段五并行 sub-agent 个数在剩余预算 < 30% 时自动降级为串行
   - 跨过 500K 立即 `yield` 终止

3. **失败回退 3 次**：
   - 单模块 sub-agent 失败累计 3 次后标 `FAILED`，不再 retry（与 ADR-0004 复用配额）
   - cross-ref 与覆盖率回退共用 3 次配额
   - 3 次预算耗尽后**强制进入 ADR-0009「预算耗尽兜底」**

4. **预算耗尽兜底**（与 ADR-0009 联动）：
   - 强制终止后输出 `analysis/STATE_REPORT.md`：
     - 已完成阶段 + 产物列表
     - 已失败阶段 + 失败原因
     - 下次重试的建议入口（从哪里恢复）
   - 用户可手动 `--resume` 从上次中断点继续

5. **预算可视化**：
   - 阶段八产出的 `analysis/README.md` 顶部增加「SLA 报告」块：
     - 实际耗时 / 时间预算（mm:ss / 30:00）
     - 实际 token / token 预算（NNNK / 500K）
     - 失败次数 / 失败预算（N / 3）
     - 超预算预警（红/黄/绿）

## Alternatives

- **G1. 保守 180min / 3M / 10**——适合大型 monorepo，但单次跑成本高。
- **G2. 平衡 90min / 1.5M / 5**——适合生产，但调试时太慢。
- **G3. 激进 30min / 500K / 3（本 ADR）**——试误友好，成本低，但大仓库可能不够。
- **G4. 动态预算**——按目标仓库 LOC 自适应，太复杂。

## Consequences

- skill 用户跑一次预期花费 $5 左右，单次等待 30 分钟以内。
- 大仓库（> 100MB 或 monorepo）可能频繁触发预算耗尽，需用户手动接力。
- ADR-0004 单模块重试 3 次耗尽后即触发本 ADR 兜底。
- §10 验收标准增加「SLA 报告在 analysis/README.md」作为可验证项。
- 阶段八 §8 README 模板更新顶部块。
- 后续可以引入「`--sla-tier conservative`」作为 v2 升级，不影响当前设计。

## Open Questions

- [ ] 30 分钟计时从 Phase-2a 启动还是从「用户调用 skill 时」开始？两种语义差几秒，影响验收。
- [ ] token 计费是否计入 tree-sitter 等本地进程？（不计入 LLM API 即可，但需要明确文档）
- [ ] 大型 monorepo 用户是否需要 `conservative` SLA tier？这可作为 v2 优化项，本 ADR 暂不实现。
