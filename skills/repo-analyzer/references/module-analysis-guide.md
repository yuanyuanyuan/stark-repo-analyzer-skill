# 模块分析指南

## 核心方法

按业务功能划分模块，不按文件或目录。一个逻辑模块可能跨越多个文件，一个文件也可能包含多个模块的部分实现。

分析深度标准：**交给另一个 AI 能仅凭报告复现系统的设计**。读者能理解模块的设计思路、职责边界、与其他模块的协作方式，并能参与架构讨论。

## 全局视角要求

**每个模块的分析都必须回答两个全局问题：**

1. **在整个项目中的角色**：这个模块为什么存在？去掉它系统会怎样？它服务于项目的哪个核心目标？
2. **与其他模块的设计协同**：它和其他模块之间的契约是什么？这种协作模式是否与项目整体的设计哲学一致？

孤立地分析模块是最常见的错误。一个模块的设计选择往往是被其他模块的约束所驱动的——不理解这些约束，就无法理解设计动机。

## 模块分析完整性四要素

每个核心模块的分析必须覆盖以下四个要素，缺任何一项都不算完整：

1. **核心数据结构** — 贴关键接口/类型定义（不是全部，只贴理解设计必需的）
2. **执行流程** — 用文字或 Mermaid 时序图描述调用链，标注源文件路径和行号
3. **设计决策** — 为什么选这个方案，不选另一个，权衡了什么
4. **模块间依赖** — 谁调用谁，数据怎么流转，共享了什么状态

> 锚点要求：四要素中的每一项判断都必须带源码锚点（`文件路径`、`文件路径:行号` 或 `文件路径:行号范围`）；没有锚点的判断只能作为假设或开放问题，不能写成确定性结论。

检验标准：如果另一个 AI 只读你的分析（不看源码），能否画出这个模块的架构图并解释它的工作原理？如果不能，说明缺了某个要素。

必须包含：业务问题、设计思路和架构模式、核心流程（Mermaid 图）、协作关系、设计权衡。
不需要包含：完整类型定义、所有函数签名、所有参数列表、错误枚举逐项说明。

## Evidence Matrix

每个核心模块草稿必须先写 Markdown Evidence Matrix，再写叙事分析。Evidence Matrix 是草稿里的证据结构，用来让主 agent 比较模块、发现缺口和合成最终报告；它不是 JSON schema，不要求自动解析，也不是硬质量门。

核心模块 Evidence Matrix 至少包含以下字段：

| 字段 | 含义 |
|------|------|
| Module Role | 模块在系统中的职责，回答为什么存在、服务哪个核心目标 |
| Entry Points | 进入该模块的入口、调用边界或外部触发点 |
| Core Data Structures | 理解设计必需的数据结构、类型或状态 |
| Main Flow | 主执行流程，以及流程中最关键的控制或数据转换 |
| Cross-Module Dependencies | 依赖谁、谁依赖它、共享什么状态或契约 |
| Key Design Decisions | 关键设计决策、取舍和替代方案代价 |
| Risk Areas | 错误处理、配置加载、插件/扩展点、缓存、并发、权限与安全边界、generated/vendor/test 边界等风险；核心模块必须包含至少一条相关风险路径抽样 |
| Source Evidence | 支撑上述判断的源码锚点或文档来源 |
| Open Questions | 当前证据无法回答、需要主 agent 交叉验证的问题 |

Evidence Matrix 必须引用或回应对应 Evidence Plan 中的架构问题。关键判断必须继承源码锚点要求；没有锚点或证据不足的内容只能写成限制、假设或开放问题。

核心模块的 Risk Areas 不能只写抽象风险标签。每条风险路径抽样必须记录：抽样对象、源码锚点、发现结果、对架构评价的影响。只抽样与模块实际相关的风险类别，不要求覆盖全部类别；不相关类别必须说明不适用原因。风险抽样是人工/流程级要求，不引入自动风险扫描器、安全扫描器、并发分析工具或硬质量门。

跨模块推断必须用【待主 agent 验证】标注，并进入 Open Questions 或等价待验证位置；在主 agent 交叉验证完成前，不得在模块草稿里写成已验证的确定性结论。这些待验证项会进入最终报告前的 Unsupported Claims 检查。

## 模块识别方法

1. **业务功能角度** — 项目提供哪些核心业务能力？
2. **数据流角度** — 数据从输入到输出经过哪些转换阶段？
3. **职责角度** — 业务需求变化时，哪些代码需要一起改？

## 分析深度

- **核心模块**（创新点、架构关键组件）：设计思路讲透、核心流程有图有解读、设计决策解释权衡、协作关系画清楚
- **次要模块**（工具函数、标准封装）：使用简化 Evidence Matrix，说明职责、入口或涉及文件、关键证据、风险或开放问题；特别之处必须有源码锚点

## Subagent 并行分析

阶段 6 必须使用 Agent 工具为每个核心模块启动独立 subagent 并行分析。

调度策略：
- 每个核心模块 → 一个独立 Agent subagent（`subagent_type: "general-purpose"`）
- 所有次要模块 → 合并到一个 Agent subagent 批量处理
- 所有 subagent 在同一消息中并行启动

## 分析结构
先写 Markdown Evidence Matrix，再用自然语言描述设计意图。默认不暴露函数名/参数名/类型定义，只有设计特别精妙时才附代码片段。

### Evidence Matrix（必须先写）

使用 Markdown 表格或分节列表均可，不要求固定排版，但必须覆盖以下语义字段：

1. Module Role — 模块角色，说明它为什么存在、服务项目哪个核心目标
2. Entry Points — 入口点、调用边界或触发点
3. Core Data Structures — 理解设计必需的数据结构、类型或状态
4. Main Flow — 主执行流程和关键控制/数据转换
5. Cross-Module Dependencies — 依赖关系、共享状态和协作契约；跨模块推断用【待主 agent 验证】标注
6. Key Design Decisions — 关键设计决策、取舍和替代方案代价
7. Risk Areas — 错误处理、配置加载、插件/扩展点、缓存、并发、权限与安全边界、generated/vendor/test 边界等风险；必须包含至少一条相关风险路径抽样，写明抽样对象、源码锚点、发现结果和对架构评价的影响
8. Source Evidence — 支撑判断的源码锚点或文档来源
9. Open Questions — 证据不足或需要主 agent 交叉验证的问题

Evidence Matrix 必须回应 Evidence Plan 中的架构问题。关键判断必须有源码锚点；没有证据的问题保留在 Open Questions，不要在叙事里写成确定性结论。

风险路径抽样要求：至少抽样一条与本模块相关的风险路径。优先从错误处理、配置加载、插件/扩展点、缓存、并发、权限与安全边界、generated/vendor/test 边界中选择与模块实际相关的类别。每条抽样必须有源码锚点，不能只写“可能有安全风险/并发风险”这类抽象标签；如果某类风险不适用，在 Risk Areas 或 Open Questions 中说明原因。

### 叙事分析

1. 在项目中的角色 — 这个模块为什么存在？去掉它系统会怎样？
2. 解决什么问题 — 业务背景，没有它系统会怎样
3. 设计思路 — 方案及理由、放弃的替代方案、核心设计模式
4. 核心数据结构 — 贴理解设计必需的关键接口/类型定义（不是全部）
5. 核心业务流程 — Mermaid 流程图 + 自然语言解读，标注源文件路径和行号
6. 与其他模块的设计协同 — 依赖谁、谁依赖它、协作方式、共享状态、这种协作模式是否与项目整体设计哲学一致。跨模块结论用【待主 agent 验证】标注
7. 关键设计决策 — 1-3 个最重要的决策及权衡（为什么选这个方案，替代方案的代价）
8. Deep Research 洞察 — 替代方案代价、业界对比、如果重新设计
9. 扩展点（如适用）
10. 亮点与问题 — 涉及文件列表

- 不要等全部文件读完再一次性写入
- 源码锚点清单（可选；行数统计仅作规模参考）在最后追加

### Subagent 协作规范

- **只分析分配的文件**，不越界
- **跨模块推断用【待主 agent 验证】标注**，主 agent 在阶段 7 交叉验证，并在阶段 8 做 Unsupported Claims 检查；未验证项不得进入最终报告的确定性叙事
- **深度优先于广度**，宁可把一个核心流程讲透
- **Evidence Plan 优先**，先回答分配计划中的架构问题，再扩展到必要的相邻证据
- **Evidence Matrix 先行**，核心模块草稿先提交统一证据结构，再写叙事分析；次要模块使用简化矩阵
- **Risk Sampling 必做**，每个核心模块至少抽样一条相关风险路径，并让发现结果影响后续批判性评价；未验证推断继续按 Unsupported Claims 规则降级
- **全局视角**，将模块放在项目整体语境中分析，解释设计选择如何服务于项目整体
- **叙事衔接**，草稿开头用 1-2 句说明本模块与前一个模块的关系，草稿结尾用 1 句铺垫下一个模块
