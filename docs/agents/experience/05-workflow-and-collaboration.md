# 协作与执行方法经验

## 1. 每项工作都要有单一产物与验证证据

**结论：** 原子任务需要固定输入、单一责任、明确输出、可执行验证和失败边界；没有验证证据的完成状态只能是待验证。

**等级：** 已核验。

**仓库证据：** [goals.txt：原子任务通过条件](../../../goals.txt)、[V1 实施目标：验证](../../goals/stark-repo-analyzer-v1-implementation.md#verification)。

## 2. 代理并发不能降低证据质量

**结论：** 当分析工作可拆分时，模块分析可并行，但来源核验、覆盖率、交叉验证与最终融合不能因此缺失。运行时不能并发时可降为串行；质量门槛不应随之降级。

**等级：** 记忆支持，产品能力边界已核验。

**来源范围：** 关于参考 skill 原生子代理分配、standard 模式报告深度和 physical run 单代理限制的项目记忆。

**仓库证据：** [repo-analyzer skill](../../../skills/repo-analyzer/SKILL.md)、[物理基线：报告保真度边界](../../baseline/physical-baseline.md#report-fidelity-boundary)。

## 3. 先做只读审计，再修改控制面

**结论：** 对 Graphify 版本迁移、模式变更或报告差异，先读取当前 CLI、doctor、契约、测试、参考基线与 ADR，列出最小修改集、风险和需要新增的测试，再进入实现。

**等级：** 记忆支持。

**来源范围：** 2026-07-13 对 Graphify 控制面与 click reference baseline 的只读审计任务。

**边界：** 审计结论必须以实际文件与测试重新核验；原始提示词不等于变更已经合入。

## 4. 用户决策要与事实记录分开

**结论：** 例如“Ruff 先跳过”“只使用开源工具”“先做小项目对比”“不要用外部编排技能”等，属于范围或方法选择；它们需要记录为决策，但不能被伪装成技术事实或验收通过。

**等级：** 记忆支持。

**来源范围：** 本项目多轮目标调整、Ruff 例外、工具选择和代理协作指示。

## 5. 经验库本身也需要可审计边界

**结论：** Meta_Kim 的 lifecycle checkpoint 适合召回上下文，但不是可直接复用的知识库。项目经验应保存结论、来源、适用范围、反例和证据等级，并将原始 observation 保留在本机记忆服务。

**等级：** 已核验本机行为；方法为本次归档决策。

**仓库证据：** [采集清单](00-meta-kim-collection-manifest.md)、[首批经验卡](2026-07-13-v1-execution-and-evidence.md)。
