# v1.1 人工验收清单（Evidence Anchor First）

> 来源：`evidence-first-v1-roadmap/v1.1-evidence-anchor-first.to-spec.md` 的 Testing Decisions。
> v1.1 不引入自动化工具，本清单为**初始人工验收**提供可执行的核对步骤。验收目标：确认 agent 的目标函数已从「读够行数」切换到「核心结论有源码锚点」，且无效阅读成本下降。

## 验收前置

- 选一个**代表性仓库**（建议含大文件 / 中等规模，例如 2–10 万行），分别用**旧流程（v1.0）**与 **v1.1** 各跑一次完整分析。
- 两次使用相同的分析模式（推荐「标准分析」）与相同的用户回答，尽量控制变量。
- 记录两次的：阅读行数（Read 行范围并集）、估算 token 消耗、最终报告路径。

## 验收项

### 1. 无效阅读成本下降 ✅/❌
- 对比旧流程与 v1.1 的**阅读行数**与 **token 消耗**，应呈下降趋势。
- 关注点：v1.1 不应为「凑覆盖率」去分段铺读低价值源码（大文件应定位关键段落精准读）。

### 2. 核心结论都有源码锚点 ✅/❌
- 通读最终报告，抽取每个核心模块的**关键判断**（角色、流程、设计决策、权衡）。
- 每个关键判断应带 `文件:行号` 或等价稳定位置描述。
- 允许未锚定的内容存在，但只能以**假设 / 观察 / 开放问题**的措辞出现，不能是确定性结论。

### 3. 核心模块未被跳过 ✅/❌
- 核对阶段 5 识别的核心模块清单，确认每个核心模块都在报告中被覆盖。
- 减少阅读**不得**导致核心模块被一句带过或遗漏（防跳过是锚点机制的保留职责之一）。

### 4. 无锚点内容已降级 ✅/❌
- 找出报告中所有未带源码锚点的论断。
- 确认它们被明确标注为假设、观察或开放问题，而非写成「项目采用 X 架构」「Y 是核心原因」这类确定性结论。

### 5. 行覆盖率仅作规模参考 ✅/❌
- 确认技能不再把行覆盖率百分比作为质量信号或阻断条件。
- 若有 `drafts/08-coverage.md`，确认其仅作规模诊断，不进入最终报告，也不影响报告产出。
- 模式表（SKILL.md / README）的列应为「源码锚点深度（分析深度指引，非质量门）」，而非覆盖率百分比目标。

### 6. 阶段与 subagent 策略不变 ✅/❌
- 确认阶段 1–8 名称与顺序未变，subagent 并行调度策略（每核心模块一个、次要合并、同消息并行启动）未变。
- 确认未引入 CLI / JSON schema / graphify / ctags / ast-grep / 硬质量门等 v1.1 Out of Scope 内容。

## 验收结论模板

```
仓库：{repo}
模式：{mode}
阅读行数：旧 {old} → v1.1 {new}（↓/↑）
token：  旧 {old} → v1.1 {new}（↓/↑）
项1 无效阅读成本下降：  ✅/❌
项2 核心结论有锚点：    ✅/❌
项3 核心模块未跳过：    ✅/❌
项4 无锚点内容降级：    ✅/❌
项5 行覆盖率仅规模参考：✅/❌
项6 阶段/subagent 不变： ✅/❌
备注：{说明}
```

验收通过条件：项 1–5 全部 ✅，且项 6 为 ✅（确保是最小改动、未越界）。
