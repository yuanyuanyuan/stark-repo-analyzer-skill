# v1.2 人工验收清单（Evidence Plan）

> 来源：`evidence-first-v1-roadmap/v1.2-evidence-plan.to-spec.md` 的 Testing Decisions。
> v1.2 不引入自动化工具，本清单为**初始人工验收**提供可执行的核对步骤。验收目标：确认 agent 在深读源码前先提出每个核心模块的架构问题和候选证据范围，让后续阅读从一开始就是问题驱动的。

## 验收前置

- 选一个**代表性仓库**（建议含 3 个以上核心模块，且存在入口、错误处理或配置等可抽查路径）。
- 使用包含 v1.2 的 repo-analyzer 跑一次标准分析。
- 保留本次分析工作目录中的 `drafts/05-modules-plan.md`、`drafts/06-module-*.md` 和最终报告。

## 验收项

### 1. 每个核心模块启动前都有 Evidence Plan ✅/❌

- 打开 `drafts/05-modules-plan.md`，核对阶段 5 识别出的每个核心模块是否都有对应 Evidence Plan。
- Evidence Plan 必须出现在启动 subagent 之前，而不是模块草稿完成后补写。

### 2. Evidence Plan 包含架构问题 ✅/❌

- 每个核心模块的 Evidence Plan 应包含要回答的架构问题。
- 架构问题应帮助判断模块角色、流程、设计决策或权衡，不能只是候选文件列表。

### 3. Evidence Plan 字段语义完整 ✅/❌

- 每个核心模块的 Evidence Plan 应包含以下语义字段：
  - 模块
  - 架构问题
  - 候选文件或候选入口
  - 必需证据类型
  - 可能需要抽查的风险路径
  - 预期输出到模块草稿的判断范围
- 不要求 Markdown 表格或列表格式逐字一致，只要求字段语义完整。

### 4. Subagent 草稿回应 Evidence Plan ✅/❌

- 打开每个 `drafts/06-module-*.md`，核对草稿是否回应对应 Evidence Plan 中的架构问题。
- 若候选文件不足、风险路径无法验证或判断范围需要收窄，草稿应标为限制、假设或开放问题。

### 5. 候选范围降低无目的阅读 ✅/❌

- 对比 Evidence Plan 的候选文件或入口与 subagent 草稿中的源码锚点。
- 草稿中的主要阅读应围绕候选入口、必需证据类型和风险路径展开，而不是按目录机械铺读。
- 若 subagent 扩展到候选范围之外，应能解释扩展原因。

### 6. v1.2 没有越界引入后续产物 ✅/❌

- 确认本次流程没有要求 Evidence Matrix、JSON schema、新 CLI、自动生成 Evidence Plan、repo map、预算档或硬质量门。
- v1.2 只实现计划层增强，并继续继承 v1.1 的源码锚点要求。
