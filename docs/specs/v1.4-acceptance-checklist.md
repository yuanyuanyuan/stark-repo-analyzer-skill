# v1.4 人工验收清单（Unsupported Claims）

> 来源：`evidence-first-v1-roadmap/v1.4-unsupported-claims.to-spec.md` 的 Testing Decisions。
> v1.4 不实现自动检测、LLM judge、硬阻塞或 quality-gate-report。本清单用于人工验收最终报告是否清楚区分已验证结论与未验证内容。

## 验收前置

- 选一个**代表性仓库**（建议含 3 个以上核心模块，并且存在跨模块依赖或开放问题）。
- 使用包含 v1.4 的 repo-analyzer 跑一次标准分析。
- 保留本次分析工作目录中的 `drafts/06-module-*.md`、`drafts/07-cross-validation.md`、`drafts/08-insights.md`（如有）和最终报告。

## 验收项

### 1. 最终报告核心确定性结论都有证据 ✅/❌

- 打开最终报告，抽取 5-10 条核心确定性结论（模块角色、主流程、关键设计决策、跨模块关系等）。
- 核对每条是否具备源码锚点、项目文档证据或外部一手来源。
- 抽查 2-3 个锚点，回到源码确认位置能支撑对应判断。

### 2. 无证据判断被降级，而不是写成事实 ✅/❌

- 检查最终报告中是否仍存在“看起来确定但没有证据”的结论。
- 没有证据的内容应出现在假设、开放问题、限制说明或 unsupported area。
- 验收关注最终报告行为，不要求内部草稿措辞逐字一致。

### 3. Evidence Matrix 开放问题进入最终合成 ✅/❌

- 打开各核心模块 Evidence Matrix 的 Open Questions。
- 核对这些问题是否进入 `drafts/07-cross-validation.md` 或最终报告。
- 若问题已解决，最终报告应能看到验证依据；若未解决，不得写成已验证结论。

### 4. 跨模块结论经过主 agent 验证 ✅/❌

- 打开模块草稿中标注【待主 agent 验证】的跨模块推断。
- 核对它们是否在交叉验证阶段被处理。
- 未验证的跨模块推断不得进入最终报告的确定性叙事。

### 5. 最终报告能区分已验证与未验证内容 ✅/❌

- 读者应能从最终报告看出哪些是已验证结论，哪些是假设/开放问题/限制说明/unsupported area。
- 不要求固定章节标题或固定 Markdown 排版，只要求语义可区分。

### 6. v1.4 没有越界引入后续产物 ✅/❌

- 确认本次流程没有要求自动断言扫描、LLM judge、硬阻塞、quality-gate-report JSON、新 CLI 或硬质量门。
- v1.4 只要求最终报告前的流程级 Unsupported Claims 检查，并继续继承 v1.1 源码锚点、v1.2 Evidence Plan 与 v1.3 Evidence Matrix。
