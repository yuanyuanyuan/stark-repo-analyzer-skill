# v1.7 人工验收清单（Markdown Repo Map）

> 来源：`evidence-first-v1-roadmap/v1.7-markdown-repo-map.to-spec.md` 的 Testing Decisions。
> v1.7 不实现 repo-map JSON、scan CLI、生态命令执行（npm/cargo/go/pip 等）、graphify 硬依赖或符号枚举器。本清单用于人工验收深读前是否先生成 Markdown Repo Map，并让它驱动 Evidence Plan 与阅读范围。

## 验收前置

- 选一个**代表性仓库**（建议含多个顶层目录、明确入口和 manifest，并且存在 test/generated/vendor 之类的低价值区域）。
- 使用包含 v1.7 的 repo-analyzer 跑一次标准分析。
- 保留本次分析工作目录中的 `drafts/02-repo-map.md`、`drafts/03-plan.md`、`drafts/05-modules-plan.md`、`drafts/06-module-*.md` 和最终报告。

## 验收项

### 1. Repo Map 存在并包含必要字段 ✅/❌

- 打开 `drafts/02-repo-map.md`。
- 检查是否覆盖目录结构、语言分布、入口候选、manifest（依赖声明文件）、核心文档、测试/generated/vendor 候选和高风险区域候选。
- 不要求 Markdown 排版逐字一致，只核对字段语义完整。

### 2. Repo Map 只写候选信号，不写最终架构结论 ✅/❌

- 检查 Repo Map 是否把内容表述为"入口候选""可能的高风险区域""疑似 generated/vendor"等候选信号。
- 不应看到模块边界定论、设计决策评价或确定性架构结论——这些应留给 Evidence Plan 和模块草稿。

### 3. Evidence Plan 引用 Repo Map 候选入口和模块信息 ✅/❌

- 打开 `drafts/05-modules-plan.md`。
- 检查每个核心模块 Evidence Plan 的候选文件或入口是否来自 Repo Map 标出的入口候选、manifest 和模块信息。
- 架构问题应贴近 Repo Map 反映的真实项目结构，而不是从 README 或目录名直接猜测。

### 4. generated/vendor/test 候选影响后续阅读范围 ✅/❌

- 对比 Repo Map 的 generated/vendor/test 候选与 `drafts/06-module-*.md`。
- 检查这些低价值区域是否被收窄或排除在核心深读之外，而不是被当作核心源码逐行阅读。

### 5. v1.7 没有越界引入 CLI 或生态命令 ✅/❌

- 确认 Repo Map 只用现有系统命令（`find`、`ls`、`wc`、`git` 等）和人工整理产出。
- 确认本次流程没有执行 npm/cargo/go/pip 等生态命令，也没有引入 repo-map JSON、scan CLI、graphify 硬依赖或符号枚举器。
- v1.7 继续继承 v1.1 源码锚点、v1.2 Evidence Plan、v1.3 Evidence Matrix、v1.4 Unsupported Claims、v1.5 Risk Sampling 与 v1.6 Budget Profiles。
