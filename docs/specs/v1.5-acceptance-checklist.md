# v1.5 人工验收清单（Risk Sampling）

> 来源：`evidence-first-v1-roadmap/v1.5-risk-sampling.to-spec.md` 的 Testing Decisions。
> v1.5 不实现自动风险识别、安全扫描器、并发分析工具、硬质量门或 quality-gate-report。本清单用于人工验收核心模块是否把风险路径纳入分析视野，并让风险发现影响最终评价。

## 验收前置

- 选一个**代表性仓库**（建议含 3 个以上核心模块，并且至少存在错误处理、配置、扩展点、缓存、并发、安全边界或 generated/vendor/test 边界中的若干类风险路径）。
- 使用包含 v1.5 的 repo-analyzer 跑一次标准分析。
- 保留本次分析工作目录中的 `drafts/05-modules-plan.md`、`drafts/06-module-*.md`、`drafts/07-cross-validation.md`、`drafts/08-insights.md`（如有）和最终报告。

## 验收项

### 1. 每个核心模块都有风险路径抽样记录 ✅/❌

- 打开每个核心模块草稿。
- 检查 Evidence Matrix 的 Risk Areas 或专门风险小节中是否至少有一条与该模块相关的风险路径抽样。
- 风险路径可以来自错误处理、配置加载、插件/扩展点、缓存、并发、权限与安全边界、generated/vendor/test 边界等类别。
- 不要求每个模块覆盖全部风险类别。

### 2. 风险抽样包含源码锚点 ✅/❌

- 每条有效风险抽样应包含抽样对象、源码锚点、发现结果和对架构评价的影响。
- 只有“可能有安全风险”“并发可能复杂”这类抽象标签，不算有效抽样。
- 抽查 2-3 个风险锚点，回到源码确认锚点能支撑对应发现。

### 3. 不适用风险类别有理由说明 ✅/❌

- 检查 Evidence Plan 或模块草稿中未抽样的显著风险类别。
- 如果某类风险与模块不相关，应看到简短理由，例如“该模块无运行时配置入口”“该模块只处理静态生成产物，不参与并发执行路径”。
- 不要求逐项列完所有类别，但不能把明显相关的风险路径静默省略。

### 4. 风险抽样进入交叉验证或合成前判断 ✅/❌

- 打开 `drafts/07-cross-validation.md`。
- 检查各核心模块的重要风险发现是否被汇总、验证、降级或保留为开放问题。
- 未验证风险推断不得在后续报告中写成确定性结论。

### 5. 最终报告批判性评价引用风险发现 ✅/❌

- 打开最终报告的评价、问题、限制或工程成熟度相关章节。
- 检查最终报告是否引用风险路径抽样结果，而不是只复述主流程或 happy path。
- 风险发现应能影响对缺陷、边界、演进成本或工程成熟度的判断。

### 6. v1.5 没有越界引入自动扫描或硬 gate ✅/❌

- 确认本次流程没有要求自动风险识别、安全扫描器、并发分析工具、硬阻塞、quality-gate-report JSON、新 CLI 或自动解析 Evidence Matrix。
- v1.5 只要求人工/流程级风险路径抽样，并继续继承 v1.1 源码锚点、v1.2 Evidence Plan、v1.3 Evidence Matrix 与 v1.4 Unsupported Claims。
