# v1.2 Evidence Plan Spec

## Problem Statement

v1.1 要求核心结论必须有源码锚点，但它仍然没有解决 subagent 在开始分析前“不知道为什么读、读什么、要证明什么”的问题。没有计划时，agent 仍可能按文件列表机械探索，只是最后补上锚点。

用户需要在深读源码前先看到每个核心模块的架构问题和候选证据范围，让后续阅读从一开始就是问题驱动的。

## Solution

v1.2 在现有模块规划阶段引入 Evidence Plan。Evidence Plan 是一个轻量 Markdown 计划，不要求 JSON，不要求 CLI，不要求自动生成。

每个核心模块在进入 subagent 分析前，必须明确：

- 该模块要回答的架构问题；
- 候选文件或候选入口；
- 必需证据类型；
- 可能需要抽查的风险路径；
- 预期输出到模块草稿的判断范围。

Evidence Plan 可以嵌入现有模块规划草稿中，避免新增复杂产物链。

## User Stories

1. As a repo-analyzer 使用者, I want 分析前看到核心模块的问题清单, so that 我知道报告会围绕什么展开。
2. As a repo-analyzer 使用者, I want agent 先规划证据再读源码, so that 阅读成本更可控。
3. As a repo-analyzer 使用者, I want 候选文件和证据类型被提前列出, so that 后续报告的覆盖边界更透明。
4. As a repo-analyzer 维护者, I want 用 Markdown 计划引入 Evidence-First 行为, so that 不需要提前实现 scan CLI。
5. As a repo-analyzer 维护者, I want Evidence Plan 复用现有模块规划阶段, so that 改动不扩大到整个 workflow。
6. As a subagent, I want 在 prompt 里收到模块问题和候选证据范围, so that 我能有目的地读取源码。
7. As a 主 agent, I want 用 Evidence Plan 分配 subagent 工作, so that 不同模块不会重复盲读。
8. As a 后续实现 agent, I want v1.2 只实现计划层, so that 它能独立于 Evidence Matrix 发布。

## Implementation Decisions

- 在模块规划阶段增加 Evidence Plan。
- Evidence Plan 采用 Markdown 表格或列表。
- Evidence Plan 的最小字段包括模块、架构问题、候选文件、必需证据类型、风险路径。
- Evidence Plan 不需要机器可读 schema。
- Evidence Plan 不自动判断架构结论，只提供阅读和分析范围。
- subagent prompt 应引用对应模块的 Evidence Plan。
- v1.2 继承 v1.1 的源码锚点要求。
- 不引入新的 CLI 或依赖。

## Testing Decisions

- 检查每个核心模块在启动 subagent 前是否有 Evidence Plan。
- 检查 Evidence Plan 是否包含架构问题，而不仅是文件列表。
- 检查 subagent 草稿是否回应了 Evidence Plan 中的问题。
- 检查候选文件范围是否减少了无目的源码读取。
- 测试不应要求 Evidence Plan 的 Markdown 格式逐字一致，只要求字段语义完整。

## Out of Scope

- 不引入 Evidence Matrix。
- 不要求 Evidence Plan 自动生成。
- 不要求 repo map。
- 不要求质量门。
- 不要求预算档生效。
- 不引入 JSON schema。

## Further Notes

v1.2 的杠杆在于把“先读再想”改为“先问再读”。它仍然很轻，但会明显改善 subagent 的阅读方向。
