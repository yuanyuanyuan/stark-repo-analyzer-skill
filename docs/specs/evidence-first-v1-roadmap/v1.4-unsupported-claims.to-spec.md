# v1.4 Unsupported Claims Spec

## Problem Statement

即使 v1.3 有了 Evidence Matrix，最终报告仍可能在合成阶段把没有证据的推断写成确定性结论。对于架构分析来说，这比遗漏信息更危险，因为用户会把未经验证的判断当作事实。

用户需要报告清楚地区分已验证结论、合理假设和开放问题，避免“看起来深刻但不可审计”的输出。

## Solution

v1.4 引入 unsupported claims 机制。最终报告前，主 agent 必须检查核心确定性结论是否有源码锚点或文档证据。缺少证据的内容不能进入确定性叙事，只能进入假设、开放问题、限制说明或 unsupported area。

v1.4 仍是人工/流程级检查，不实现自动 gate。

## User Stories

1. As a repo-analyzer 使用者, I want 无证据结论被显式标记, so that 我不会把推测当事实。
2. As a repo-analyzer 使用者, I want 报告区分已验证和未验证内容, so that 我能知道哪些地方需要继续读源码。
3. As a repo-analyzer 使用者, I want 开放问题集中呈现, so that 我能快速判断分析边界。
4. As a repo-analyzer 维护者, I want 最终报告前增加 unsupported claims 检查, so that 报告可信度提升。
5. As a repo-analyzer 维护者, I want 该机制保持人工检查, so that 不需要提前实现质量门工具。
6. As a 主 agent, I want 合成报告时检查证据状态, so that 我不会放大 subagent 的推断。
7. As a subagent, I want 跨模块推断被标记为待验证, so that 主 agent 能集中交叉验证。
8. As a 后续实现 agent, I want unsupported claims 先成为流程规则, so that v1.8 再把它纳入软质量门。

## Implementation Decisions

- 最终报告前必须检查核心确定性结论的证据状态。
- 没有源码锚点、文档证据或外部一手来源的结论不得写成确定性结论。
- 未验证内容必须移动到开放问题、限制说明或 unsupported area。
- 跨模块推断必须在主 agent 交叉验证前保持待验证状态。
- Evidence Matrix 中的开放问题不能在最终报告中静默消失。
- v1.4 不实现自动检测。
- v1.4 不要求机器可读质量报告。

## Testing Decisions

- 抽查最终报告中的核心结论，确认每条都有证据。
- 检查没有证据的判断是否被降级为开放问题或限制说明。
- 检查 Evidence Matrix 的开放问题是否被带入最终合成阶段。
- 检查跨模块结论是否经过主 agent 验证。
- 测试重点是最终报告行为，而不是检查内部草稿措辞。

## Out of Scope

- 不实现 quality-gate-report。
- 不自动扫描报告断言。
- 不引入 LLM judge。
- 不引入硬阻塞。
- 不改变报告结构的写作风格。

## Further Notes

v1.4 是可信度版本。它不显著减少 token，但能显著降低错误结论进入最终报告的概率。
