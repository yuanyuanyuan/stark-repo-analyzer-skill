# v1.7 Markdown Repo Map Spec

## Problem Statement

即使有 Evidence Plan，agent 仍需要先理解仓库结构才能提出好问题。如果没有仓库地图，LLM 容易从 README 或少量目录名直接推断模块边界，导致候选文件不准、核心入口遗漏或重复探索。

用户需要一个低依赖、低成本的 repo map，让 agent 在深读前先获得项目结构、语言分布、入口候选和排除区域。

## Solution

v1.7 引入 Markdown Repo Map。它使用现有系统命令和人工整理生成，不要求新增 CLI，不要求 JSON schema，不执行生态命令。

Repo Map 作为 Evidence Plan 的输入，包含目录结构、语言分布、入口候选、manifest、核心文档、测试/generated/vendor 候选和可能的高风险区域。

## User Stories

1. As a repo-analyzer 使用者, I want agent 深读前先生成仓库地图, so that 分析范围更准确。
2. As a repo-analyzer 使用者, I want repo map 标出入口和 manifest, so that 报告能从真实项目结构出发。
3. As a repo-analyzer 使用者, I want generated/vendor/test 候选被提前标出, so that agent 不会把低价值区域当核心源码读。
4. As a repo-analyzer 维护者, I want 用现有系统命令生成 repo map, so that 不增加安装门槛。
5. As a repo-analyzer 维护者, I want repo map 先用 Markdown 表达, so that 后续验证字段稳定后再 JSON 化。
6. As a 主 agent, I want 用 repo map 生成 Evidence Plan, so that 架构问题更贴近仓库事实。
7. As a subagent, I want repo map 提供模块背景, so that 我能理解自己负责的文件在全局中的位置。
8. As a 后续实现 agent, I want v1.7 不执行生态命令, so that 分析扫描保持只读、低风险。

## Implementation Decisions

- 分析前生成 Markdown Repo Map。
- Repo Map 包含目录结构、语言分布、入口候选、manifest、核心文档、测试/generated/vendor 候选和高风险区域候选。
- Repo Map 只能提供候选信号，不能直接写最终架构结论。
- Repo Map 使用现有系统命令和人工整理。
- 不执行生态命令。
- 不要求 JSON schema。
- 不引入 scan CLI。
- Repo Map 应作为 Evidence Plan 的输入。

## Testing Decisions

- 检查 Repo Map 是否存在并包含必要字段。
- 检查 Repo Map 是否只写候选信号，不写最终架构结论。
- 检查 Evidence Plan 是否引用 Repo Map 的候选入口和模块信息。
- 检查 generated/vendor/test 候选是否影响后续阅读范围。
- 测试不应断言具体命令，只检查 Repo Map 的外部内容。

## Out of Scope

- 不实现 repo-map JSON。
- 不实现 scan CLI。
- 不执行 npm、cargo、go、pip 等生态命令。
- 不引入 graphify 依赖。
- 不引入符号枚举器。
- 不做完整语义索引。

## Further Notes

v1.7 是 v1.x 从纯 prompt 规则走向轻量确定性辅助的第一步。它必须保持低侵入，否则会提前滑向 v2.0 的 breaking 工具链。
