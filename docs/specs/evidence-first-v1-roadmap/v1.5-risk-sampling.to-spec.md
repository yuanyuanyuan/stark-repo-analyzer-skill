# v1.5 Risk Sampling Spec

## Problem Statement

Evidence-First 分析如果只沿着主流程寻找证据，可能会产出流畅但片面的报告。真实架构质量往往体现在错误处理、配置、插件、缓存、并发、安全边界、generated/vendor/test 边界等风险路径中。

用户需要报告不仅说明系统正常如何工作，也要说明系统在边界条件和风险路径下如何表现。

## Solution

v1.5 引入风险路径抽样。每个核心模块必须识别并抽样至少一个与该模块相关的风险路径。风险路径存在时，模块草稿需要记录抽样对象、证据锚点、发现结果和对架构评价的影响。

v1.5 不要求覆盖所有风险类别，也不要求自动识别风险路径。它只要求每个核心模块把风险路径纳入分析视野。

## User Stories

1. As a repo-analyzer 使用者, I want 报告覆盖风险路径, so that 我不会只看到 happy path。
2. As a repo-analyzer 使用者, I want 知道模块如何处理错误和配置, so that 我能评估工程成熟度。
3. As a repo-analyzer 使用者, I want 风险发现影响最终评价, so that 批判性分析更真实。
4. As a repo-analyzer 维护者, I want 用抽样方式控制风险分析成本, so that 深度提升不会无限扩大范围。
5. As a repo-analyzer 维护者, I want 风险抽样仍保持文档规则, so that 不需要提前实现自动风险扫描。
6. As a subagent, I want prompt 中要求风险路径抽样, so that 我不会只分析主流程。
7. As a 主 agent, I want 汇总风险抽样结果, so that 最终报告能诚实评价缺陷和边界。
8. As a 后续实现 agent, I want v1.5 与预算档分离, so that 风险抽样先验证有效性，再由 v1.6 调整强度。

## Implementation Decisions

- 每个核心模块必须执行风险路径抽样。
- 风险类别包括错误处理、配置加载、插件/扩展点、缓存、并发、权限与安全边界、generated/vendor/test 边界。
- 只抽样与模块实际相关的风险类别。
- 每个抽样结果必须包含抽样对象、源码锚点、发现结果和影响说明。
- 不存在相关风险类别时，必须说明为什么不适用。
- 风险抽样结果应进入 Evidence Matrix 或模块草稿的专门小节。
- 最终报告的批判性评价应引用风险抽样结果。

## Testing Decisions

- 检查每个核心模块是否有风险抽样记录。
- 检查风险抽样是否包含源码锚点。
- 检查风险抽样是否影响最终报告评价，而不是只停留在草稿中。
- 检查不适用风险类别是否有理由说明。
- 测试不要求覆盖全部风险类别，只要求存在合理抽样。

## Out of Scope

- 不自动识别风险路径。
- 不要求风险类别全覆盖。
- 不引入安全扫描器。
- 不引入并发分析工具。
- 不实现硬质量门。

## Further Notes

v1.5 的价值在于避免 Evidence-First 退化成 Main-Path-First。它增加少量阅读成本，但通常能显著提升报告的批判性和可信度。
