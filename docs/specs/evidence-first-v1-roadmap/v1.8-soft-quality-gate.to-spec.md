# v1.8 Soft Quality Gate Spec

## Problem Statement

v1.1 到 v1.7 已经引入源码锚点、Evidence Plan、Evidence Matrix、unsupported claims、风险抽样、预算档和 Repo Map。但如果最终报告前没有统一检查点，这些规则可能散落在草稿中，主 agent 合成时仍可能遗漏。

用户需要一个轻量质量门，在不引入硬阻塞和 CLI 的情况下，让最终报告前的放行标准可见。

## Solution

v1.8 引入 Soft Quality Gate。主 agent 在最终报告前生成 Markdown 质量门摘要，检查 Evidence Matrix 完整性、源码锚点、风险抽样、unsupported claims、预算执行和 Repo Map 使用情况。

Soft Quality Gate 不阻塞分析完成，但所有未通过项必须进入最终报告的开放问题、限制说明或 unsupported area。

## User Stories

1. As a repo-analyzer 使用者, I want 最终报告前有质量门摘要, so that 我能知道报告哪些部分可信。
2. As a repo-analyzer 使用者, I want 未通过项进入限制说明, so that 报告不会隐藏分析缺口。
3. As a repo-analyzer 使用者, I want 质量门覆盖证据、风险和预算, so that 报告质量更透明。
4. As a repo-analyzer 维护者, I want 先实现软质量门, so that 可以验证 gate 规则是否合理。
5. As a repo-analyzer 维护者, I want 质量门仍是 Markdown, so that 不需要提前实现 JSON gate。
6. As a 主 agent, I want 合成前有放行清单, so that 我能系统检查草稿缺口。
7. As a subagent, I want 知道草稿会被质量门检查, so that 我会更稳定地提交 Evidence Matrix。
8. As a 后续实现 agent, I want v1.8 的质量门规则可迁移到 v2.0, so that 后续 CLI gate 有现实依据。

## Implementation Decisions

- 最终报告前必须生成 Markdown 质量门摘要。
- 质量门检查 Evidence Matrix 完整性、源码锚点、风险抽样、unsupported claims、预算执行和 Repo Map 使用情况。
- 每个检查项必须标记通过、未通过或不适用。
- 未通过项必须有原因和后续处理位置。
- 未通过项不硬阻塞报告生成。
- 最终报告必须反映未通过项。
- v1.8 不要求 JSON。
- v1.8 不引入 gate CLI。

## Testing Decisions

- 检查每次分析是否生成质量门摘要。
- 检查质量门摘要是否覆盖全部必需检查项。
- 检查未通过项是否进入最终报告的限制说明或开放问题。
- 检查质量门是否引用实际草稿和证据，而不是空泛打勾。
- 测试不应要求硬阻塞行为。

## Out of Scope

- 不实现硬质量门。
- 不实现 quality-gate-report JSON。
- 不自动解析 Evidence Matrix。
- 不使用 LLM judge 自动评分。
- 不阻止最终报告生成。

## Further Notes

v1.8 的目标是让质量缺口可见，而不是让流程不可运行。它为 v2.0 的硬 gate 提供规则样本和失败案例。
