# v1.1 Evidence Anchor First Spec

## Problem Statement

repo-analyzer 当前把分析质量主要绑定到 Read 行覆盖率。这个规则能防止 agent 完全跳过核心模块，但在大仓库里会诱导 agent 为了凑百分比读取大量低价值源码。用户真正需要的是可信的架构结论，而不是“读了很多行”的过程指标。

v1.1 要解决的第一性问题是：先把 agent 的目标函数从“读够多少行”改为“核心结论是否有源码锚点”。这是 v2.0 Evidence-First 路线中最小、最直接、最高杠杆的第一刀。

## Solution

v1.1 将行覆盖率从主质量目标降级为规模参考，把源码锚点提升为核心质量要求。

核心行为变为：每个核心模块的关键判断必须带 `文件:行号` 或等价源码位置；没有源码锚点的内容不能写成确定性架构结论。agent 仍然可以读取必要源码，但读取目的必须是支撑架构问题和设计判断，而不是满足行数比例。

v1.1 保持纯文档 skill 形态，不引入 CLI、JSON schema、graphify、ctags、ast-grep 或硬质量门。

## User Stories

1. As a repo-analyzer 使用者, I want 核心结论都有源码锚点, so that 我可以判断报告是否可信。
2. As a repo-analyzer 使用者, I want agent 不再为了凑行覆盖率读取大文件, so that 分析耗时和 token 成本下降。
3. As a repo-analyzer 使用者, I want 快速分析也能保留关键证据, so that 速度提升不会变成空泛摘要。
4. As a repo-analyzer 使用者, I want 行数统计只作为规模参考, so that 分析重点回到架构问题。
5. As a repo-analyzer 维护者, I want 用最小文档改动改变 agent 行为, so that 不必提前引入 v2.0 的复杂工具链。
6. As a repo-analyzer 维护者, I want 保留现有阶段和报告结构, so that v1.1 可以独立发布并快速验证。
7. As a subagent, I want 明确知道结论必须锚定源码, so that 我不会提交没有证据的模块判断。
8. As a 主 agent, I want 用源码锚点检查模块草稿, so that 最终报告不会混入无依据断言。

## Implementation Decisions

- 行覆盖率不再作为主质量门。
- 行数统计仍可用于估算仓库规模和辅助选择分析模式。
- 每个核心模块必须包含源码锚点支撑的关键判断。
- 源码锚点可以是文件路径、行号、行号范围或等价的稳定位置描述。
- 无源码锚点的内容只能作为假设、观察或开放问题。
- 现有阶段 1-8 保持不变。
- 现有 subagent 并行策略保持不变。
- 不新增任何运行时依赖。
- 不生成新的机器可读产物。

## Testing Decisions

- 测试外部行为，而不是测试 prompt 文案逐字匹配。
- 用同一个代表性仓库对比旧流程和 v1.1：阅读行数或 token 是否下降。
- 检查最终报告的核心结论是否都有源码锚点。
- 检查核心模块是否仍被覆盖，不能因为减少阅读而跳过核心模块。
- 检查没有源码锚点的内容是否被降级为假设或开放问题。
- 人工验收可以作为 v1.1 的初始验证方式，因为本版本不引入自动化工具。

## Out of Scope

- 不引入 Evidence Plan。
- 不引入 Evidence Matrix。
- 不引入风险路径抽样。
- 不引入预算档重定义。
- 不生成 repo map。
- 不实现质量门。
- 不试点关键单元覆盖率。
- 不要求 graphify、ctags 或 ast-grep。

## Further Notes

v1.1 的价值不在于产物更完整，而在于立即纠正错误目标函数。如果这一版不能降低无效阅读成本，后续工具化也很难真正获得收益。
