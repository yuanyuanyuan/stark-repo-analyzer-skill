# v1.6 Budget Profiles Spec

## Problem Statement

当前快速、标准、深度模式主要围绕覆盖率比例表达，但当行覆盖率不再是主质量指标后，模式之间需要新的差异定义。否则 agent 仍可能在快速模式里过度分析，或在深度模式里缺少明确加深方向。

用户需要每个分析模式对应可理解、可执行、可验收的 evidence 深度和成本边界。

## Solution

v1.6 将快速、标准、深度模式重定义为 evidence 数量、风险抽样强度、报告长度、subagent 数量和外部调研强度的组合。

模式差异不再由“读多少行”决定，而由“回答多少关键问题、提供多少证据、抽样多少风险、输出多深报告”决定。

## User Stories

1. As a repo-analyzer 使用者, I want 快速模式成本低且边界清楚, so that 我能快速了解项目全貌。
2. As a repo-analyzer 使用者, I want 标准模式保持完整架构分析, so that 我能用于常规学习和评估。
3. As a repo-analyzer 使用者, I want 深度模式明确增加风险和权衡分析, so that 我知道额外成本花在哪里。
4. As a repo-analyzer 使用者, I want 每次分析记录预算目标和实际执行, so that 我可以比较不同模式收益。
5. As a repo-analyzer 维护者, I want 模式差异从行覆盖率迁移到 evidence 深度, so that 新目标函数保持一致。
6. As a repo-analyzer 维护者, I want 预算档先用文档规则表达, so that 不需要精确 token 计量。
7. As a subagent, I want prompt 中有证据读取预算, so that 我知道应该深入到什么程度。
8. As a 主 agent, I want 超预算时缩小 scope 或说明原因, so that 分析过程可控。

## Implementation Decisions

- 快速模式聚焦核心路径和核心模块，风险抽样最小，报告较短。
- 标准模式覆盖核心模块、关键边界和主要设计决策，风险抽样标准。
- 深度模式扩展到次要模块、边缘路径和替代方案，风险抽样增强。
- 每个模式必须声明 evidence 深度、风险抽样强度、报告长度、subagent 数量上限和外部调研强度。
- 每次分析必须记录预算目标和实际执行摘要。
- 超预算时必须缩小范围、降低深度或显式说明原因。
- v1.6 不要求精确 token 统计。
- v1.6 继承 v1.1 到 v1.5 的全部行为。

## Testing Decisions

- 用同一仓库分别运行快速、标准、深度模式，比较输出差异。
- 检查快速模式是否减少 evidence 数量和报告长度。
- 检查标准模式是否保留核心模块和关键权衡。
- 检查深度模式是否增加风险路径和替代方案分析。
- 检查每次分析是否记录预算目标和实际执行摘要。
- 测试不应要求精确 token 数，只检查相对行为和产物边界。

## Out of Scope

- 不实现自动 token 预算中断。
- 不要求精确计费统计。
- 不引入性能 benchmark harness。
- 不改变最终报告的核心章节质量要求。
- 不引入 CLI。

## Further Notes

v1.6 让成本控制第一次成为 workflow 的显式契约。它是 v1.x 中“收益可见性”最强的版本之一，因为用户能直接比较不同模式的投入产出。
