# v1.3 Evidence Matrix Spec

## Problem Statement

v1.2 让分析前有了 Evidence Plan，但模块草稿仍可能以散文形式输出。散文草稿难以比较、难以合并，也难以检查是否缺少入口点、主流程、跨模块依赖或风险证据。

用户需要每个模块草稿都有统一的证据结构，让主 agent 能稳定融合多个 subagent 的结果，并在最终报告前发现缺口。

## Solution

v1.3 引入 Evidence Matrix。每个核心模块的 subagent 草稿必须包含统一的 Evidence Matrix，用于记录模块角色、入口点、核心数据结构、主流程、跨模块依赖、关键设计决策、风险点、源码证据和开放问题。

Evidence Matrix 在 v1.3 中仍然是 Markdown 结构，不要求 JSON，不要求自动 gate。它的目标是让模块草稿更可比较、更可审计，而不是立即机器化。

## User Stories

1. As a repo-analyzer 使用者, I want 每个模块分析都有统一证据结构, so that 我可以快速判断报告是否完整。
2. As a repo-analyzer 使用者, I want 模块角色、入口、主流程和权衡分开呈现, so that 我能理解设计而不是只看代码说明。
3. As a repo-analyzer 使用者, I want 开放问题被保留, so that 不确定性不会被伪装成结论。
4. As a repo-analyzer 维护者, I want subagent 草稿格式更稳定, so that 主 agent 合并成本下降。
5. As a repo-analyzer 维护者, I want Evidence Matrix 先以 Markdown 出现, so that 后续可以验证字段是否合适再 JSON 化。
6. As a subagent, I want 明确的 Evidence Matrix 字段, so that 我的输出不会遗漏关键架构要素。
7. As a 主 agent, I want 用 Evidence Matrix 对比模块草稿, so that 我可以发现跨模块矛盾和缺口。
8. As a 后续实现 agent, I want v1.3 不实现 gate, so that 结构化输出和质量门可以分开验证。

## Implementation Decisions

- 每个核心模块草稿必须包含 Evidence Matrix。
- Evidence Matrix 的最小字段为模块角色、入口点、核心数据结构、主流程、跨模块依赖、关键设计决策、风险点、源码证据、开放问题。
- Evidence Matrix 使用 Markdown 表格或分节列表。
- Evidence Matrix 必须引用 v1.2 Evidence Plan 中的问题。
- Evidence Matrix 中的关键判断必须继承 v1.1 的源码锚点要求。
- 次要模块可以使用简化 Evidence Matrix。
- 不要求 JSON 格式。
- 不实现自动质量门。

## Testing Decisions

- 检查每个核心模块草稿是否包含 Evidence Matrix。
- 检查 Evidence Matrix 是否覆盖所有必需字段。
- 检查源码证据字段是否包含锚点。
- 检查开放问题是否被保留而不是丢弃。
- 检查主 agent 是否能基于 Evidence Matrix 合成最终报告。
- 测试应关注字段语义完整性，不关注 Markdown 具体排版。

## Out of Scope

- 不实现 module-evidence JSON。
- 不实现 quality gate。
- 不引入关键单元覆盖率。
- 不要求自动抽查 judgment 质量。
- 不新增工具依赖。

## Further Notes

v1.3 是从“更会读”走向“更会交付证据”的关键版本。它为后续 unsupported claims、风险抽样和质量门提供结构基础。
