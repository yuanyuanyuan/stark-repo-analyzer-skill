# v1.9 Key Unit Coverage Pilot Spec

## Problem Statement

v2.0 的核心设想是用关键代码单元覆盖率替代行覆盖率。但如果在没有试点的情况下直接把关键单元覆盖率做成硬 schema 和硬 gate，风险很高：关键单元如何定义、如何列分母、如何判断已锚定，都可能在真实仓库中暴露问题。

用户需要在 v1.x 阶段先验证关键单元覆盖率是否真的比行覆盖率更能解释报告质量。

## Solution

v1.9 试点关键单元覆盖率。它使用轻量人工/命令辅助清单记录核心模块的关键单元、已锚定分析状态、未覆盖原因和覆盖摘要。

v1.9 不要求 ctags、ast-grep、graphify、稳定单元 id 或硬质量门。它的目标是验证模型，而不是完成 v2.0 的完整实现。

## User Stories

1. As a repo-analyzer 使用者, I want 覆盖率反映关键设计单元而不是源码行数, so that 我能更好判断报告质量。
2. As a repo-analyzer 使用者, I want 未覆盖关键单元有原因, so that 我知道报告边界在哪里。
3. As a repo-analyzer 使用者, I want 关键单元覆盖摘要可审计, so that 覆盖率不是一个不透明数字。
4. As a repo-analyzer 维护者, I want 先试点关键单元覆盖率, so that v2.0 的 schema 不会凭空设计。
5. As a repo-analyzer 维护者, I want 不要求符号枚举器, so that 试点不增加安装门槛。
6. As a 主 agent, I want 用关键单元清单检查模块分析完整性, so that 我能发现被忽略的入口、数据结构或状态定义。
7. As a subagent, I want 知道关键单元需要被锚定分析, so that 我不会只写模块叙事。
8. As a 后续实现 agent, I want v1.9 产出 v2.0 schema 的经验, so that doctor、units 和 gate 的实现更稳。

## Implementation Decisions

- 引入关键单元覆盖率试点。
- 关键单元可以包括入口点、核心数据结构、公共 API、状态定义、配置定义和关键跨模块依赖。
- 关键单元清单可以使用 Markdown 或轻量 JSON。
- 每个关键单元记录是否已锚定分析、源码锚点、设计判断和未覆盖原因。
- 覆盖率用于审计和学习，不作为硬门控。
- 不要求稳定单元 id。
- 不要求自动枚举分母。
- 不要求符号枚举器。
- v1.9 结果用于决定 v2.0 的 coverage-units schema 是否合理。

## Testing Decisions

- 在代表性仓库上产出关键单元清单。
- 检查关键单元是否覆盖入口点、核心数据结构、状态/配置和关键依赖。
- 检查已覆盖单元是否同时具备源码锚点和设计判断。
- 检查未覆盖单元是否有原因。
- 对比关键单元覆盖率和旧行覆盖率，判断哪个更能解释报告可信度。
- 测试不应要求自动枚举完全准确，只验证试点是否提供更好的审计信号。

## Out of Scope

- 不实现 coverage-units JSON 硬 schema。
- 不实现 units CLI。
- 不实现 gate CLI。
- 不要求 ctags 或 ast-grep。
- 不要求 graphify。
- 不要求关键单元 id 稳定。
- 不把关键单元覆盖率作为硬阻塞。

## Further Notes

v1.9 是 v2.0 之前最重要的验证版本。如果关键单元覆盖率在轻量试点中无法明显优于行覆盖率，就不应该在 v2.0 中把它做成硬门控。v1.9 的结果应该直接反馈到 v2.0 的 `coverage-units` schema、doctor 检查项和 gate 规则。
