# v1.9 人工验收清单（Key Unit Coverage Pilot）

> 来源：`evidence-first-v1-roadmap/v1.9-key-unit-coverage-pilot.to-spec.md` 的 Testing Decisions。
> v1.9 不实现 coverage-units JSON 硬 schema、units CLI、gate CLI、ctags、ast-grep、graphify 硬依赖或硬阻塞。本清单用于人工验收关键单元覆盖率试点是否提供比行覆盖率更好的审计信号，并为 v2.0 沉淀反馈。

## 验收前置

- 选一个**代表性仓库**（建议含 3 个以上核心模块，且具有明确入口点、核心数据结构、状态/配置定义）。
- 使用包含 v1.9 的 repo-analyzer 跑一次标准分析。
- 保留本次分析工作目录中的 `drafts/06-module-*.md`、`drafts/07-cross-validation.md` 和最终报告。

## 验收项

### 1. 核心模块草稿包含关键单元清单 ✅/❌

- 打开每个 `drafts/06-module-*.md`（核心模块）。
- 检查是否包含关键单元清单，覆盖入口点、核心数据结构、公共 API、状态/配置定义和关键跨模块依赖中与该模块相关的单元。
- 不要求 Markdown 排版逐字一致，只核对字段语义完整。

### 2. 已覆盖单元同时具备锚点和设计判断 ✅/❌

- 检查清单中标记为已覆盖（✅）的关键单元是否同时包含：
  - 源码锚点（`文件路径` 或 `文件路径:行号`）
  - 设计判断摘要（一句话说明该单元的架构意义）
- 只有锚点没有判断、或只有判断没有锚点的，不算有效覆盖。

### 3. 未覆盖单元有原因说明 ✅/❌

- 检查清单中标记为未覆盖（❌）的关键单元是否有明确的未覆盖原因。
- 未覆盖单元不能静默省略，也不能只写"未覆盖"而不说明原因。

### 4. 关键单元覆盖情况已汇总到交叉验证阶段 ✅/❌

- 打开 `drafts/07-cross-validation.md`。
- 检查是否包含各模块关键单元覆盖情况汇总：已覆盖单元数、未覆盖单元数及原因。
- 检查是否有对"关键单元覆盖率是否优于行覆盖率作为可信度信号"的初步判断（哪怕只是一句话的观察）。

### 5. 试点结论沉淀为 v2.0 反馈 ✅/❌

- 检查 `drafts/07-cross-validation.md` 或 `drafts/08-insights.md` 中是否包含对 v2.0 的反馈：
  - 关键单元类型定义是否合理
  - 关键单元覆盖率是否比行覆盖率更能解释报告可信度
  - 对 v2.0 `coverage-units` schema、doctor 检查项或 gate 规则的可执行建议
- 不要求结论一定为正面；如果试点发现关键单元覆盖率并不优于行覆盖率，也应记录。

### 6. v1.9 没有越界引入符号枚举器或硬阻塞 ✅/❌

- 确认本次流程没有要求 ctags、ast-grep、graphify、稳定单元 id、自动枚举分母、units CLI、gate CLI 或把关键单元覆盖率作为硬阻塞。
- v1.9 继续继承 v1.1-v1.8 的全部行为。
