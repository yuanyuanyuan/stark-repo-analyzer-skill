# v1.3 人工验收清单（Evidence Matrix）

> 来源：`evidence-first-v1-roadmap/v1.3-evidence-matrix.to-spec.md` 的 Testing Decisions。
> v1.3 不引入 JSON、自动解析、新 CLI 或硬质量门。本清单用于人工验收 Markdown Evidence Matrix 是否让模块草稿更可比较、更可审计，并能支撑最终报告合成。

## 验收前置

- 选一个**代表性仓库**（建议含 3 个以上核心模块，并且存在入口、跨模块依赖或风险路径）。
- 使用包含 v1.3 的 repo-analyzer 跑一次标准分析。
- 保留本次分析工作目录中的 `drafts/05-modules-plan.md`、`drafts/06-module-*.md`、`drafts/07-cross-validation.md` 和最终报告。

## 验收项

### 1. 每个核心模块草稿都有 Evidence Matrix ✅/❌

- 打开每个核心模块的 `drafts/06-module-*.md`。
- 核对草稿是否先写 Markdown Evidence Matrix，再进入叙事分析。
- Evidence Matrix 可以是表格或分节列表，不要求固定排版。

### 2. Evidence Matrix 必需字段语义完整 ✅/❌

- 每个核心模块 Evidence Matrix 应包含以下语义字段：
  - 模块角色
  - 入口点
  - 核心数据结构
  - 主流程
  - 跨模块依赖
  - 关键设计决策
  - 风险点
  - 源码证据
  - 开放问题
- 不要求 Markdown 格式逐字一致，只要求字段语义完整。

### 3. Evidence Matrix 回应 Evidence Plan ✅/❌

- 打开 `drafts/05-modules-plan.md`，找到每个核心模块对应的 Evidence Plan。
- 核对 Evidence Matrix 是否引用或回应 Evidence Plan 中的架构问题。
- 如果候选证据不足、风险路径无法验证或判断范围需要收窄，Evidence Matrix 应保留为限制、假设或开放问题。

### 4. 源码证据字段包含锚点 ✅/❌

- Evidence Matrix 的关键判断应有 `文件路径`、`文件路径:行号` 或等价源码锚点。
- 没有源码锚点的关键判断不能写成确定性结论，必须降级为假设、观察、限制或开放问题。
- 抽样 2-3 个核心模块锚点，回到源码确认锚点位置能支撑对应判断。

### 5. 开放问题被保留而不是丢弃 ✅/❌

- 核对 Evidence Matrix 中的 Open Questions 是否进入 `drafts/07-cross-validation.md` 的交叉验证或限制说明。
- 如果开放问题最终被解决，最终报告应能看到验证依据。
- 如果开放问题没有被解决，最终报告不得把它写成已验证结论。

### 6. 次要模块使用简化 Evidence Matrix ✅/❌

- 打开 `drafts/06-module-secondary.md`。
- 核对次要模块是否使用简化 Evidence Matrix 或等价结构，至少包含职责、入口或涉及文件、关键证据、风险或开放问题。
- 次要模块的特别判断仍需源码锚点；简化规则不能削弱核心模块要求。

### 7. 主 agent 能基于 Evidence Matrix 合成最终报告 ✅/❌

- 打开 `drafts/07-cross-validation.md` 和最终报告。
- 核对主 agent 是否使用 Evidence Matrix 对比模块角色、入口、主流程、跨模块依赖、风险点和开放问题。
- 最终报告应保留 Why > What 的叙事深度，不应退化为 Evidence Matrix 字段拼接。

### 8. v1.3 没有越界引入后续产物 ✅/❌

- 确认本次流程没有要求 `module-evidence/*.json`、JSON schema、新 CLI、自动解析 Evidence Matrix、自动 quality gate 或硬质量门。
- v1.3 只实现 Markdown Evidence Matrix，并继续继承 v1.1 源码锚点和 v1.2 Evidence Plan。
