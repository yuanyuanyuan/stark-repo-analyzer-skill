# v1.6 人工验收清单（Budget Profiles）

> 来源：`evidence-first-v1-roadmap/v1.6-budget-profiles.to-spec.md` 的 Testing Decisions。
> v1.6 不实现精确 token 统计、自动 token 中断、性能 benchmark harness 或新 CLI。本清单用于人工验收快速、标准、深度模式是否真正改变 evidence 深度、风险抽样强度和报告边界。

## 验收前置

- 选一个**代表性仓库**（建议含 3 个以上核心模块，并且至少有若干次要模块、边界路径或替代方案可分析）。
- 使用包含 v1.6 的 repo-analyzer 分别跑快速、标准、深度三种模式。
- 保留每次分析工作目录中的 `drafts/03-plan.md`、`drafts/05-modules-plan.md`、`drafts/06-module-*.md`、`drafts/07-cross-validation.md`、`drafts/08-insights.md`（如有）和最终报告。

## 验收项

### 1. 每次分析记录预算目标 ✅/❌

- 打开 `drafts/03-plan.md`。
- 检查是否记录本次模式，以及对应的 evidence 深度、风险抽样强度、报告长度目标、subagent 数量上限和外部调研强度。
- 预算目标应是可观察的工作流边界，不要求精确 token 数。

### 2. 模式选择影响 Evidence Plan 与 subagent 输入 ✅/❌

- 打开 `drafts/05-modules-plan.md` 和核心模块 subagent prompt。
- 检查 Evidence Plan 是否体现当前模式的候选入口、架构问题数量、风险路径候选和判断范围。
- 检查 subagent prompt 是否包含当前模式的证据读取预算，而不是只写“快速/标准/深度”名称。

### 3. 快速模式减少 evidence 数量和报告长度 ✅/❌

- 对比三种模式的 `drafts/06-module-*.md` 和最终报告。
- 快速模式应聚焦核心路径和核心模块，减少次要模块展开、外部调研和报告篇幅。
- 快速模式仍必须保留核心结论源码锚点、Evidence Plan、Evidence Matrix、风险抽样和 Unsupported Claims 规则。

### 4. 标准模式保留核心模块和关键权衡 ✅/❌

- 标准模式应覆盖核心模块、关键边界和主要设计决策。
- 检查核心模块草稿是否有完整 Evidence Matrix，并能解释主要权衡和架构评价。
- 标准模式应是默认的完整架构分析，不应退化成快速摘要。

### 5. 深度模式增加风险路径和替代方案分析 ✅/❌

- 深度模式应比标准模式增加次要模块、边缘路径、替代方案或工程成熟度判断。
- 检查风险抽样是否更充分，例如同一核心模块抽样多条风险路径，或覆盖边缘路径/替代方案风险。
- 深度模式增加的内容仍需源码锚点或可审计证据；无证据内容必须降级。

### 6. 每次分析记录实际执行摘要 ✅/❌

- 打开 `drafts/07-cross-validation.md` 或 `drafts/08-insights.md`。
- 检查是否记录实际 subagent 数量、核心/次要模块覆盖情况、风险抽样数量、外部调研范围、报告长度是否符合目标。
- 若超预算，应说明如何缩小范围、降低深度、合并 subagent、减少调研或保留开放问题。

### 7. v1.6 没有越界引入自动 token 或新工具链 ✅/❌

- 确认本次流程没有要求精确 token 计费、自动 token 中断、预算 CLI、benchmark harness、JSON gate 或硬阻塞。
- v1.6 只要求人工/流程级预算档，并继续继承 v1.1 源码锚点、v1.2 Evidence Plan、v1.3 Evidence Matrix、v1.4 Unsupported Claims 与 v1.5 Risk Sampling。
