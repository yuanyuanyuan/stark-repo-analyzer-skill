# 真实UAT回归测试规则（v2.1 · 独立 `codex exec` 严格执行 skill）

更新时间：2026-07-11  
关联：Issue #12 · ticket 12 · 用户纠正「docs-only / 同会话代跑」不足以称为「真实UAT回归测试」  
证据目录约定：`测试证据/v2.1-human/`（或用户指定的其它 `测试证据/v2.x-*` 目录）

## 0. 正式名称

| 字段 | 值 |
|---|---|
| **正式名称** | **真实UAT回归测试** |
| 英文/标签 | `real-uat-regression` · `codex-exec-skill-uat` |
| 规则路径 | `docs/specs/v2.1-codex-exec-uat.md` |
| 默认证据目录 | `测试证据/v2.1-human/` |
| 入口引用 | 根目录 `AGENTS.md` →「真实UAT回归测试」 |
| **Dev 分档规则（quick/standard/deep）** | [`dev-rules/real-uat-regression/`](../../dev-rules/real-uat-regression/) |

在 PR、验收报告、RUN_LOG、UAT 状态文件中，凡指本规则定义的测试，**必须**使用正式名称「真实UAT回归测试」，不要仅写含糊的「UAT」或「exec UAT」。

同会话 docs-only、同会话手写 report、同会话 multi-agent 实验 **都不是**「真实UAT回归测试」。

## 1. 为什么需要这条规则

下列验收**不等于**「真实UAT回归测试」：

| 做法 | 可证明什么 | 不能单独当作真实UAT回归测试 |
|---|---|---|
| 同会话 agent 读 `ACCEPTANCE_RESULT` / 勾选 checklist | 文档口径是否诚实 | 真实 skill 执行路径 |
| 主线程直接调 `node bin/repo-analyzer.js …` 并手写 report | CLI 机械链路 | 「编程 Agent 按 SKILL.md 跑完整分析」 |
| 同会话并行 Python worker 写 `subagent-artifacts` | 可审计的并行取证实验 | **独立进程** `codex exec` 冷启动执行 skill |
| 仅 `npm test` / typecheck | 回归 | 真实目标仓 UAT |

「真实UAT回归测试」的强制形态是：

> **新开一个单独的 `codex exec` 进程**，提示词要求 **严格执行** 仓库内  
> `skills/repo-analyzer/SKILL.md`，对真实目标仓做分析，并把报告/工件输出到指定证据目录。

## 2. 强制形态（Canonical）

### 2.1 命令模板

在**业务仓库根**（本仓库）或任意可访问 skill 路径的 shell 中执行：

```bash
codex exec "严格执行 $(pwd)/skills/repo-analyzer/SKILL.md 分析 /tmp/Long_screenshot_splitting_tool ，输出报告到 $(pwd)/测试证据/v2.1-human"
```

可替换参数：

| 占位 | 含义 | 示例 |
|---|---|---|
| `$SKILL` | skill 文件绝对路径 | `…/skills/repo-analyzer/SKILL.md` |
| `$REPO` | 被分析目标仓库 | `/tmp/Long_screenshot_splitting_tool` |
| `$OUT` | 本轮 UAT 证据输出目录（须在 `测试证据/` 下新建，不覆盖历史） | `…/测试证据/v2.1-human` |

推荐通用形：

```bash
codex exec "严格执行 $SKILL 分析 $REPO ，输出报告到 $OUT"
```

### 2.2 硬性要求

1. **独立进程**：必须 `codex exec`（或等价非交互一次性 exec），**禁止**仅用「当前 ticket 交付会话里顺便分析」顶替「真实UAT回归测试」。  
2. **严格执行 SKILL.md**：提示词必须含「严格执行」+ skill 的**绝对路径**；agent 须走 v2 强制工作流（doctor → scan → summarize → units → Evidence Plan → 模块分析 → gate → 仅在放行后合成最终报告）。  
3. **真实目标仓**：默认 `/tmp/Long_screenshot_splitting_tool`（或用户指定的其它真实路径/URL）；禁止只对 fixture 冒充真实UAT回归测试。  
4. **输出目录新建**：如 `测试证据/v2.1-human`；**不得**覆盖 `测试证据/v2.0`、`测试证据/v2.1/standard` 等历史轮次。  
5. **诚实并行**：运行时若支持 subagent，standard/deep 应按 skill 并行并写 `parallelism: active` + 分工/产物/融合；若不支持，必须写 `parallelism: degraded`，且 **不得** 标 multi-agent 完整通过。  
6. **不绕过 gate**：`allowed_to_synthesize: false` 时**禁止**生成或伪造成品 `ANALYSIS_REPORT.md` 冒充通过。

### 2.3 建议提示词正文（可整段粘贴）

```text
严格执行 {SKILL_ABS_PATH}
分析 {REPO_ABS_PATH}
输出报告到 {OUT_ABS_PATH}

约束：
1. 完整遵守 skill 中的 v2 强制工作流与质量门，不得跳过 doctor/units/gate。
2. 在 {OUT_ABS_PATH} 写入全部可审计工件（doctor/scan/units、evidence-plan、module-evidence、report、quality-gate-report；gate 通过才写 ANALYSIS_REPORT）。
3. Evidence Plan 如实记录 parallelism: active 或 degraded，以及子代理分工/产物/融合（若 active）。
4. 最终在 {OUT_ABS_PATH}/UAT_EXEC_SUMMARY.md 用中文写：命令、开始结束时间、模式、parallelism、gate 是否放行、产物清单、失败原因。
5. 若 core unparsed 非空：必须执行 Unparsed File Read Pass，并在 {OUT_ABS_PATH} 落盘补读产物（unparsed-file-reviews* 和/或 module-evidence.unparsed_manual_reads）；报告对高影响 unparsed 写 manual-read 锚点，禁止只列路径。
6. 不要修改被分析仓库源码；不要覆盖 {OUT_ABS_PATH} 以外的历史测试证据目录；不要把补读伪装成 parse_rate 提升。
```

## 3. 通过标准（真实UAT回归测试专用）

### 3.1 过程通过（真实UAT回归测试 · 过程有效）

同时满足：

- [ ] 存在可复核的 **独立** `codex exec` 调用记录（shell 历史、CI 日志或 `UAT_EXEC_SUMMARY.md` 中的完整命令）  
- [ ] `$OUT` 下出现 skill 合同内的关键工件（至少：`doctor-report.json`、`repo-map.json`、`coverage-units.json`、`evidence-plan.md`、`quality-gate-report.json`、叙事 `report.md` 或 skill 等价草稿）  
- [ ] `UAT_EXEC_SUMMARY.md`（或等价）说明模式、parallelism、gate 结论  
- [ ] 未把 degraded 写成 multi-agent 完整通过；未在 gate 失败时伪造最终通过报告  
- [ ] **core unparsed 非空时的补读产物**：若 `coverage-units.json` 中存在 core 模块 unparsed 文件，则 `$OUT` 必须出现以下至少其一：`unparsed-file-reviews/*`、`unparsed-file-reviews.json`、任一 `module-evidence/*.json` 的 `unparsed_manual_reads[]`、或 Evidence Plan「Unparsed File Read Pass」节；**禁止**仅 Unsupported 路径列表  
- [ ] 报告对高影响 unparsed（如 App/上传/导出编排）若已补读，须有 **manual-read** 锚点叙述（`文件:行号` + confidence），且 **不得** 把补读写成 parse_rate 提升或 analyzed unit  

### 3.2 产品「分析完整通过」（更严，可单独判定）

在 3.1 基础上还须：

- [ ] `quality-gate-report.json` 中 `allowed_to_synthesize: true`  
- [ ] 存在 gate 通过后生成的最终 `ANALYSIS_REPORT.md`  
- [ ] standard/deep 若声称 multi-agent 完整通过：Evidence Plan 为 `parallelism: active` 且含分工/产物/融合（见 `docs/specs/v2.0-multi-agent-acceptance.md`）  

**允许**出现：过程通过（3.1）但产品未完整通过（3.2）——例如 parse_rate 不足导致 gate 失败。此时 UAT 结论应写：

> 真实UAT回归测试过程有效；产品分析未完整通过（原因：…）。

### 3.3 失败 / 无效（真实UAT回归测试）

任一成立则「真实UAT回归测试」**无效或失败**（不得对外写「真实UAT回归测试已通过」）：

- 未使用独立 `codex exec`，仅同会话手写报告  
- 未引用/未执行 `skills/repo-analyzer/SKILL.md` 工作流  
- 输出写进错误目录或覆盖历史证据且未声明  
- gate 失败却声称完整通过，或静默跳过 doctor/gate  
- core unparsed 非空但无补读产物，仅 Unsupported 路径列表却声称过程有效/完整通过  
- 把补读成功写成 parse_rate 提升或 analyzed unit  

## 4. 与其它证据目录的关系

| 目录 | 角色 |
|---|---|
| `测试证据/v2.0/` | 历史 v2.0 证据 |
| `测试证据/v2.1/standard/` | 同会话 CLI + 串行分析实验（degraded） |
| `测试证据/v2.1/standard-multiagent/` | 同会话多进程取证实验（active 实验） |
| **`测试证据/v2.1-human/`** | **真实UAT回归测试默认落盘**（独立 `codex exec` 严格执行 skill） |
| `测试证据/v2.0/pr-evidence/` | PR 文档口径 / unit 证据包，**不能替代**真实UAT回归测试 |

> 同会话实验有价值，但**只有**独立 `codex exec` 轮次才计入「真实UAT回归测试」。

## 5. 真实UAT回归测试 · 执行检查清单（给人 / agent）

1. 确认目标仓路径存在且 commit 已知  
2. `mkdir -p` 新的 `$OUT`（如 `测试证据/v2.1-human`）  
3. 运行 §2.1 的 `codex exec …`  
4. 等待 exec 结束，检查 `$OUT` 工件与 `UAT_EXEC_SUMMARY.md`  
5. 若 `coverage-units.json` 显示 core unparsed 非空：检查补读产物与 report 中 manual-read 锚点（见 §3.1）  
6. 按 §3 判定：过程通过 / 产品完整通过 / 失败  
7. 将命令、exit code、结论链到 PR 或 `测试证据/v2.1/RUN_LOG.md`  

## 6. 维护

- 规则文件路径：`docs/specs/v2.1-codex-exec-uat.md`（产品层：独立 exec 定义与过程/产品分层）
- **按模式执行的 dev 分档规则**：`dev-rules/real-uat-regression/`（quick / standard / deep）
- `AGENTS.md` 必须同时指向 `dev-rules/real-uat-regression/` 与本文件（见仓库根 `AGENTS.md`「真实UAT回归测试」节）
- 修改 skill / gate / 需求时：**同步更新** `dev-rules/real-uat-regression/` 与本文件模板提示词是否仍有效
