# v2.2 Beats-v1.0：Insight Probe、Degraded Delivery 与验收锁

> 来源：grill-with-docs 共识（CONTEXT.md + docs/adr/0001–0009）  
> 状态：ready-for-agent  
> 范围：第一交付切片 P1+P2+P3（P4 deep reference Tooling Debt 另开）

## Problem Statement

Primary Reader（Human Decision Reader + Agent Consumer）需要一次仓库架构分析后，拿到**可决策、可继续自动执行**的终稿。对照 v1.0 与当前 2.1 真实 UAT 报告后发现：

1. **准确度退步**：v1.0 能抓住的高价值一致性硬伤（UI 承诺未进执行路径、规则多源冲突、配置/实现双写或死实现），2.1 常漏掉；coverage 与章节齐全不能代替这些洞见。
2. **有用性分裂**：2.1 强化了 Acceptance Auditor 的可审计工厂，却弱化了给人读的可行动密度；Agent Consumer 也缺少稳定可解析的探针结论与 delivery 状态。
3. **交付合同错误对齐验收者**：gate 红（`allowed_to_synthesize=false`）时禁止写 `ANALYSIS_REPORT.md`，导致 deep 等工作流跑完后主用户空手；「诚实未放行」被当成对主用户可接受的终点。
4. **无法证明压过 v1.0**：缺少 Gold Sample、Reader Rubric 与 Agent Smoke，只能靠印象比较。

本阶段 **Beats-v1.0 Bar** 只强制三维：有用、质量、准确。速度不纳入必达；允许更慢。Quality-over-Speed。

## Solution

在现有 evidence-first 流水线（doctor→scan→units→evidence→report→gate→合成）上增加三块，组成第一交付切片 **P1+P2+P3**：

1. **P1 Insight Probe**  
   跨仓库通用探针目录（首批三类）：  
   - UI Promise → Runtime Path  
   - Multi-Source Rules  
   - Config Dual-Write / Dead Implementation  
   执行模型：确定性候选枚举 + LLM 判定；落盘 `insight-probes.json`；流程缺产物/缺类别进入 Mechanical Gate；`miss` 不挡门；`hit` 必须进入终稿风险与改造优先级。

2. **P2 Delivery Contract**  
   统一终稿路径 `ANALYSIS_REPORT.md`：  
   - Full：`delivery_status=full`（gate 全绿）  
   - Degraded：同路径，`delivery_status=degraded` + 文首横幅 + 失败门禁 + 结论边界（gate 未放行仍交付）  
   废止「gate 红禁止任何 ANALYSIS_REPORT」；「产品分析完整通过」仍只认 full。

3. **P3 验收锁**  
   - Accuracy Bar：Gold Sample Set v1（G1–G4）必中且进终稿  
   - Usefulness/Quality Bar：Reader Rubric + Agent Smoke  
   多 PR 串联落地，三包齐备前不得宣称 Beats-v1.0。

P4（deep Graphify/reference 接线债）并行另开，不阻塞本切片；不得靠放宽 reference-quality 装绿或永久 degraded 冒充 deep 完整通过。

## User Stories

1. As a Human Decision Reader, I want 一份带主链叙事与改造优先级的终稿, so that 我能据此改代码或做 onboarding，而不只看到门禁日志。
2. As an Agent Consumer, I want 稳定的 `delivery_status`、锚点与探针结论字段, so that 我能自动开 issue、改代码或二次推理。
3. As a Primary Reader, I want gate 失败时仍拿到 Degraded 终稿, so that 分析过程不白跑、我不空手。
4. As a Primary Reader, I want Degraded 终稿有显著降级标记, so that 我不会把未完整通过当成完整通过。
5. As an Acceptance Auditor, I want 「产品分析完整通过」仍绑定 Full Delivery, so that degraded 文件存在不能骗过 UAT。
6. As a Human Decision Reader, I want 报告抓住 UI 承诺与真实执行不一致, so that 我能修「界面骗人」类缺陷。
7. As a Human Decision Reader, I want 报告抓住同一规则多处定义且冲突, so that 我能收敛单一事实源。
8. As a Human Decision Reader, I want 报告抓住配置双写或未挂主路径的平行实现, so that 我能去掉死实现与分叉配置。
9. As an Agent Consumer, I want 每类 Insight Probe 都有 hit/miss/n_a 结论, so that 我不会把「没写」当成「没问题」。
10. As an Acceptance Auditor, I want 缺少 `insight-probes.json` 或缺少类别时 Mechanical Gate 失败, so that 探针流程不能被跳过。
11. As an Acceptance Auditor, I want 探针 status=miss 不导致 gate 失败, so that 没挖到硬伤的健康仓仍可 Full Delivery。
12. As a skill maintainer, I want 探针类别跨仓库通用且不绑定示例仓业务名词, so that 产品不会过拟合长截图 demo。
13. As a skill maintainer, I want 确定性步骤只枚举候选入口、由 LLM 判定硬伤, so that 跨语言脆正则不会冒充准确度。
14. As a Human Decision Reader, I want hit 必须出现在终稿风险与改造优先级, so that 洞见不会只埋在 JSON 里。
15. As an Agent Consumer, I want `insight-probes.json` 含 category/status/summary/anchors/report_ref, so that 我能机械消费。
16. As a Primary Reader, I want n_a 必须带仓库形态理由, so that 纯库/无 UI 仓不会被静默跳过或乱报。
17. As an Acceptance Auditor, I want Gold Sample G1–G3 在新版本上必中, so that 准确度相对 v1.0 不回归。
18. As an Acceptance Auditor, I want Gold Sample G4 验证非 UI 形态, so that 探针通用性有护栏。
19. As a skill maintainer, I want Reader Rubric 固定对照 v1.0, so that 有用性/质量不靠印象分。
20. As an Agent Consumer, I want Agent Smoke 能解析终稿关键字段, so that 双主用户合同可测。
21. As a skill maintainer, I want 第一交付切片含 P1+P2+P3, so that 半成品不能宣称压过 v1.0。
22. As a skill maintainer, I want 多 PR 串联而非巨型单 PR, so that 可审可回滚。
23. As a skill maintainer, I want P4 deep reference 债另开, so that 不阻塞主用户交付合同修复。
24. As an Acceptance Auditor, I want deep 未还清 Tooling Debt 时不得宣称完整通过, so that 诚实性不被永久 degraded 稀释。
25. As a Primary Reader, I want 速度不是本阶段过线条件, so that 系统不会为刷时长砍探针。
26. As a skill author, I want SKILL.md 与 UAT 规格同步 Synthesis Rule, so that 文档与实现不再「gate 红禁止写终稿」。
27. As a gate maintainer, I want Probe Process Gate 成为 quality-gate-report 的一项检查, so that 流程失败可审计。
28. As a report author, I want Full 与 Degraded 使用同一文件名, so that agent 默认路径稳定。
29. As a Human Decision Reader, I want 终稿减少模板重复灌水, so that 分析质量体现在洞见而非章节堆砌。
30. As a benchmark owner, I want Beats-v1.0 验收只绑定有用/质量/准确三维, so that 与 Quality-over-Speed 一致。
31. As a fixture test author, I want 用工作目录工件合同测试 gate 与交付状态, so that 不依赖真实 LLM 调用也能锁机械行为。
32. As a UAT runner, I want 真实UAT 区分「过程有效 / Full Delivery / Degraded Delivery」, so that 摘要不会把有 degraded 稿写成完整通过。
33. As a domain modeler, I want 术语与 ADR 与实现一致, so that Primary Reader、Delivery Contract 等词不漂移。
34. As a future implementer of P4, I want 本规格明确 reference 接线是债不是本切片范围, so that 范围不膨胀。
35. As a Primary Reader on standard mode, I want 三类探针默认启用, so that standard 不再系统性漏 v1.0 级硬伤。
36. As a Primary Reader on deep mode, I want 同等探针义务 + 可用时 Full 否则 Degraded, so that deep 失败时我仍有稿可读。

## Implementation Decisions

### 术语与决策源

- 使用根目录 `CONTEXT.md` 词汇：Primary Reader、Delivery Contract、Insight Probe、Probe Obligation、Probe Process Gate、Full/Degraded Delivery、Gold Sample、Accuracy/Usefulness/Quality Bar、Beats-v1.0 Bar、Tooling Debt、Synthesis Rule 等。
- 遵守 ADR 0001–0009；实现不得与其静默冲突。

### 工作包与节奏

- **P1** Insight Probe：Catalog、候选枚举辅助、LLM 判定、结构化产物、命中进终稿、Probe Process Gate。
- **P2** Delivery：Synthesis Rule；`ANALYSIS_REPORT.md` + `delivery_status`；skill/UAT 文案同步。
- **P3** 验收：Gold Sample Set v1、Reader Rubric、Agent Smoke 清单与（尽量）机械检查。
- **PR 节奏**：建议 PR1=P1 → PR2=P2 → PR3=P3；三包齐备前不得宣称 Beats-v1.0。
- **P4** 另票：deep reference 接线；本规格不实现放宽阈值。

### Insight Probe Catalog（首批）

| ID | Category | 意图 |
|---|---|---|
| C1 | `ui_promise_runtime_path` | 用户可感知选项/行为是否进入真实执行路径 |
| C2 | `multi_source_rules` | 同一规则是否多处定义且冲突 |
| C3 | `config_dual_write_dead_impl` | 多配置源或未挂主路径平行实现 |

每类结论：`hit | miss | n_a`；`n_a` 必须含形态理由。

### 产物合同

**`insight-probes.json`**（工作目录根，agent/确定性辅助共同保证存在）：

```json
{
  "version": 1,
  "mode": "standard",
  "probes": [
    {
      "category": "ui_promise_runtime_path",
      "status": "hit",
      "summary": "…",
      "anchors": ["path:line"],
      "report_ref": "风险或改进章节中的稳定锚点/标题",
      "candidates_considered": 3
    }
  ]
}
```

- Catalog 每一类至少一条。
- `hit` ⇒ 终稿风险/改造优先级必须出现对应内容；`report_ref` 可被检查或抽检。

**`ANALYSIS_REPORT.md`**：

- Front matter 或等价元数据必须含 `delivery_status: full|degraded`。
- `degraded` 时文首可见横幅；列出失败 Mechanical Gate 项；受影响结论降级为假设/Unsupported 边界。
- `full` 时不得带降级横幅冒充。

**`quality-gate-report.json`**：

- 新增检查 id：`insight-probe-process`（名称以实现为准，语义：产物存在 + 类别齐全 + 字段合法）。
- `status=miss` 不 fail。
- `allowed_to_synthesize` 语义保持「是否允许 **Full** Delivery」；全部 pass 才为 true。
- **合成行为**（skill 合同，可部分非 CLI）：  
  - `allowed_to_synthesize=true` → Full 终稿  
  - `false` → 仍必须写 Degraded 终稿（同路径），不得宣称完整通过  

### 确定性候选枚举（辅助，非定罪）

- 提供可复用的候选收集能力（脚本或 skill 强制步骤）：例如表单/选项字段名、重复规则符号、多份 config/平行 *Manager 实现候选。
- 输出可供 LLM 消费的候选列表；**不得**仅因命中正则就写 `hit`。

### Skill / UAT 文档

- 改写 `skills/repo-analyzer/SKILL.md` Phase 合成规则与工件表。
- 改写 `docs/specs/v2.1-codex-exec-uat.md`（或后续 UAT SSOT）：区分过程有效、Full、Degraded；删除「gate 红禁止任何 ANALYSIS_REPORT」绝对句，替换为 Synthesis Rule。
- 真实 UAT 摘要字段建议包含 `delivery_status`。

### 模块级改动方向（不写死路径细节）

- **gate 模块**：增加 Probe Process Gate；保持其余 Mechanical Gate。
- **skill 叙事与合成**：强制探针阶段、终稿引用、Full/Degraded 合成。
- **可选 CLI 辅助**：候选枚举子命令非必须；若增加，须只读目标仓。
- **测试夹具**：扩展现有 gate/e2e 夹具，构造有/无 `insight-probes.json`、full/degraded 元数据场景。

### Gold Sample Set v1

- **G1** 对齐 C1：UI Promise 未进 runtime（可用既有对照仓实例，类别名保持通用）。
- **G2** 对齐 C2：Multi-Source Rules。
- **G3** 对齐 C3：Config Dual-Write / Dead Implementation。
- **G4** 非 UI 形态仓：合法 `n_a`+理由或真实 hit；禁止静默跳过。

### Reader Rubric（有用性/质量，相对 v1.0 不退步）

至少包括：主链是否说清；改造是否可执行且有锚点；双主用户可消费；无模板同义重复灌水；探针 hit 进优先级；Unsupported/降级边界清楚。

### Agent Smoke

机械或脚本可检查：`delivery_status` 可解析；`insight-probes.json` 可解析；每类有 status；hit 带 anchors；终稿存在。

## Testing Decisions

### 什么叫好测试

- 只测**外部行为与工件合同**，不测 LLM 是否“聪明”。
- 优先最高 seam：工作目录落盘产物 + `repo-analyzer gate` 退出语义 + JSON 字段。
- 不引入真实网络/真实目标大仓作为单元测试必依赖；真实 UAT 另走 `测试证据/` 与 UAT 规格。

### 主测试缝（Seams）

**唯一主缝：分析工作目录的工件合同（Artifact Contract Seam）**  
通过现有 CLI/`gate` 与 skill 约定的文件读写观察行为：

| 观察点 | 断言方向 |
|---|---|
| `insight-probes.json` | 缺文件/缺类别 → `insight-probe-process` fail → `allowed_to_synthesize=false` |
| 合法 miss 全集 | 不因 miss fail；在其他门绿时可 full |
| `quality-gate-report.json` | 含新检查项与 reasons |
| Full 合成条件 | 全绿 ⇒ 允许 full 元数据 |
| Degraded 合成合同 | 门红时 skill/UAT 要求仍存在带 `delivery_status=degraded` 的 `ANALYSIS_REPORT.md`（CLI 若只负责 gate，则 gate 测试锁门禁；合成规则用文档+验收/e2e 夹具锁） |
| Agent Smoke | 最小解析脚本或测试读字段 |

**次缝（仅当主缝不够）：** 候选枚举纯函数对夹具仓库的候选列表稳定性（仍不判定 hit）。

**不采用的缝：** 快照整份散文报告正文；mock 内部私有函数；以 coverage 百分比代替探针。

### 优先测试模块

- `gate` 单测（扩展 `test/gate.test.js` 模式）：探针流程门、与 `allowed_to_synthesize` 关系。
- e2e/夹具链：在 prepareArtifacts 风格下附加 probes 文件。
- 可选：轻量 `insight-probes` schema 校验测试。
- 验收层：Gold Sample / Rubric 清单文档 + 真实 UAT 回归（非纯单元）。

### 先验

- 现有 `test/gate.test.js`、`test/e2e.test.js`、`test/helpers.js` createFixture 流水线。
- 现有 doctor reference 探针测试（P4 相关，本切片不扩展放宽逻辑）。

## Out of Scope

- 速度必胜 v1.0 或 wall-clock 硬 KPI。
- P4：Graphify → units `refs_status` 接线、放宽 deep `reference-quality` 阈值。
- 纯确定性自动「定罪」跨语言硬伤。
- 仓库专用探针当主机制；绑定长截图业务为唯一产品范围。
- 改变 Primary Reader 为 Acceptance Auditor。
- 恢复 quick 模式或大范围 coverage 阈值重做。
- 多 PR 合并策略的具体 GitHub branch 命名（实现自定）。
- 完整 LLM judge 自动打 Rubric 分（本切片以清单 + 机械 smoke 为主）。

## Further Notes

- 对照证据目录（问题背景）：`测试证据/v1.0没改造前` vs `real-uat-standard-20260711-1855` / `real-uat-deep-20260711-1855`。
- deep UAT 曾因 reference-quality 无 `ANALYSIS_REPORT.md`：本规格用 Degraded Delivery 修复主用户空手，**不**把该失败算 Full。
- 实现时若 `allowed_to_synthesize` 名称易误导，可在报告中另书 `full_delivery_allowed` 别名，但须保持兼容或迁移说明。
- 第一刀建议顺序仍按 P1→P2→P3 编码，便于审阅；完成定义是三包齐备而非单个 PR。
