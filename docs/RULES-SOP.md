# 规则优先级与执行 SOP

> 目的：统一“读哪些规则、谁压谁、冲突怎么判、误写指令怎么拦”的顺序，让后续每次执行都有可遵循的固定流程，不靠临场发挥，也不因一句写错的指令就绕过项目护栏。

---

## 0. 适用范围

- 本仓库（stark-repo-analyzer-skill）的所有对话、技能调用、代码改动、文档生成。
- 适用对象：本助手在本工作空间内的所有自动化决策。
- 本文件是“规则的优先级与执行方式”的元规则；具体业务规则以 `AGENTS.md` / `CLAUDE.md` 及全局设定为事实源，并按本文第 8 节分类维护。

---

## 1. 规则来源与权威链（从高到低）

| 层级 | 类别 | 性质 |
|---|---|---|
| **Tier 0** | 安全 / 合规 / 隐私硬规则 | 不可覆盖，冲突即拒绝 |
| **Tier 1** | 用户当前指令 + hooks 反馈 | 本次消息目标；hook 反馈视为用户输入 |
| **Tier 2** | 被调用的技能（skills） | 具体任务的“方法”配方 |
| **Tier 3** | 仓库内规则文件（AGENTS.md / CLAUDE.md） | 项目级常设约束 |
| **Tier 4** | 记忆 | 工作区记忆 > 用户级 MEMORY.md > 云端画像（只读，下次会话被覆盖） |
| **Tier 5** | 全局设定（settings.json） | `~/.workbuddy/settings.json` 与 `~/.codebuddy/settings.json` 共存、各司其职 |
| **Tier 6** | 智能体 / 专家团队智能体 | 执行载体，非更高规则源 |

**基本原则**：上层压下层；同层按“目标（Tier1）＞方法（Tier2）＞护栏（Tier3）”的维度理解，而非简单二选一。

---

## 2. 仓库规则三级分类：MUST / SHOULD / MAY

 Markdown 里的规则本质都是“软指引”，可被明确指令推翻。为防误写绕过，先给每条规则定级：

| 级别 | 含义 | 能否被用户指令推翻 | 约束力来源 |
|---|---|---|---|
| **MUST** | 不变量，违反即出错或违法 | **不能**（聊天层拒绝豁免） | 下沉到 hook / sandbox / Tier0 |
| **SHOULD** | 强约定，默认遵守 | **能，但须明确 + 留痕** | md 软指引 + 推翻时确认提示 |
| **MAY** | 默认偏好，非关键 | 能，无需确认 | 直接遵从 |

分类判定标准：
- 涉及安全、合规、数据不可恢复损失、区域表述规范的 → **MUST**。
- 项目约定俗成、影响一致性但不致命的（语言、必用工具、流程顺序）→ **SHOULD**。
- 纯风格、可随意调整者 → **MAY**。

## 3. 优先级裁决规则

### 3.1 总表

| 冲突双方 | 裁决 | 理由 |
|---|---|---|
| Tier1 vs Tier2 | **用户赢** | 技能是达成目标的手段，不是老板 |
| Tier1 vs Tier3(SHOULD) | **用户赢（可临时豁免）** | 用户可为此任务推翻强约定，须明确+留痕 |
| Tier1 vs Tier3(MUST) | **MUST 赢** | 不变量聊天层不可豁免 |
| Tier2 vs Tier3 | **分维度** | 技能定“怎么做”，仓库定“必须始终满足的底线”；方法服从目标，护栏框住产出 |
| 三层同时冲突 | **用户为纲、技能为手段、MUST 为硬下限** | 用户明确豁免的 SHOULD 不适用，未豁免的常设约束仍守 |

### 3.2 维度说明（避免误判）

- **Tier1 = 目标（goal）**：你要什么。
- **Tier2 = 方法（method）**：这个任务“怎么做”的最具体配方。
- **Tier3 = 护栏（guardrail）**：不论目标与方法，产出必须满足的常设约束。

所以当“用户 A / 技能 B / 仓库 C”同时出现：优先满足 A；用 B 的方法实现 A（不矛盾时）；C 中用户明确豁免的部分不适用，未豁免的 MUST 仍守住。

---

## 4. 覆盖（override）四规则

任何对 SHOULD/MUST 级仓库规则的推翻，必须满足：

1. **明确性**：仅当用户指令**清晰且具体指向某条规则**时才算豁免（如“这次跳过 graphify”）。泛泛冲突、顺手写的句子不自动豁免。
2. **留痕**：每次豁免 SHOULD/MUST，必须显式提示“已按你的要求豁免 X 仓库规则”，让豁免可见、可回溯，而非静默失效。
3. **疑似误写先确认**：指令与 SHOULD 冲突但**看起来像笔误 / 无心之失**（如随手写出与“简体中文”矛盾的句子）→ 暂停并确认，不替用户拍板。这正是挡住“写错指令”的闸。
4. **MUST 不走聊天豁免**：MUST 级只由 hook / sandbox / Tier0 拦截，聊天层直接拒绝豁免并说明。

---

## 5. 硬护栏下沉原则

- **软指引（md）可被明确指令推翻；真护栏必须在环境层**：
  - 提交拦截：`user-prompt-submit` hook
  - 会话停止：`Stop` hook（本仓库已有 review-gate）
  - 会话结束：`SessionEnd` hook（本仓库已有 lifecycle）
  - 文件系统边界：`settings.json` 的 `sandbox` 写权限
  - 不可谈判底線：Tier0 安全/合规/隐私
- **本仓库现状**：当前仅 `Stop`、`SessionEnd` 两个 hook，**无 `user-prompt-submit` 提交钩**。因此 SHOULD 级目前靠对话层（第 4 节）兜底；MUST 级靠环境与 Tier0 兜底。
- **改进方向**：把 MUST 级不变量做成环境级 hook，使其不被任何聊天指令绕过（见第 9 节维护）。

## 6. 执行 SOP（每次处理请求的固定流程）

```
□ 1. 解析用户指令（Tier1 目标）
□ 2. 是否命中技能？是 → 加载该技能工作流（Tier2 方法）
□ 3. Tier0 安全闸：是否触碰安全/合规/隐私/区域规范？是 → 拒绝并说明，流程终止
□ 4. 按权威链裁决冲突（Tier1 > Tier2 > Tier3 > Tier4 > Tier5；Tier6 为载体）
□ 5. 检查与仓库规则（Tier3）的冲突：
       - 命中 MUST → 拒绝聊天豁免，提示走环境拦截
       - 命中 SHOULD：
            · 用户指令明确推翻 → 遵从 + 留痕提示
            · 疑似误写 → 暂停确认
            · 未提及 → 默认遵守
□ 6. 执行（必要时派生智能体/专家，Tier6）
□ 7. 收尾：
       - 改了代码 → 跑 `graphify update .`
       - 有任何 SHOULD/MUST 豁免 → 在回复中显式标注
□ 8. 拿不准 / 真矛盾且无用户明示 → 问用户，不替其决定
```

---

## 7. 冲突上升规则

- 同一层内两条规则直接矛盾且用户未明示 → 停下来问，不二选一。
- 技能禁止项恰好是仓库 MUST 要求 → 以仓库 MUST 为准并适配，同时提示用户该矛盾。
- 任何涉及 Tier0 的张力 → 直接拒，不进入下层裁决。

## 8. 本仓库规则清单（单一事实源，按级维护）

> 来源：`AGENTS.md`、`CLAUDE.md`、全局设定、环境 hook。新增/变更规则时同步更新本表与对应文件。

### 8.1 MUST（不变量，环境级兜底）

| 规则 | 来源 | 约束力 |
|---|---|---|
| 安全/合规/内容安全边界 | Tier0 | 拒绝并说明 |
| 个人文件高危操作保护（桌面/下载/文档等递归删、无备份删等） | Tier0 | 拒绝并说明 |
| 区域规范：香港/澳门/台湾属中国，须按官方表述 | Tier0 | 拒绝并说明 |
| 禁止删除 `.workbuddy/` 文件夹（项目数据，非缓存） | 系统 + 环境 | 拒绝 |
| `sandbox` 写权限边界（`settings.json` extraAllowWrite） | 全局设定 | 环境强制 |

### 8.2 SHOULD（强约定，可明确豁免+留痕）

| 规则 | 来源 | 备注 |
|---|---|---|
| 生成/编辑文档用简体中文（专业术语除外） | AGENTS.md | 用户可要求改英文，须留痕 |
| 代码类问题先跑 `graphify query`（graph.json 存在时） | AGENTS.md/CLAUDE.md | 用户可要求改用 grep，须留痕 |
| dirty graphify 文件不是跳过 graphify 的理由 | AGENTS.md | — |
| `wiki/index.md` 优先于裸读源码做导航 | AGENTS.md/CLAUDE.md | — |
| `GRAPH_REPORT.md` 仅用于架构总览或上下文不足时 | AGENTS.md/CLAUDE.md | — |
| 改动代码后跑 `graphify update .` | AGENTS.md/CLAUDE.md | 收尾步骤 |
| 用户输入 `/graphify` 时调用 `skill:graphify` | AGENTS.md | 触发机制 |

### 8.3 MAY（默认偏好）

| 规则 | 来源 |
|---|---|
| 非关键的文档风格/排版默认 | 隐含 |

### 8.4 环境 hook（Tier5/环境级）

| Hook | 位置 | 作用 |
|---|---|---|
| Stop → review-gate | `~/.codebuddy/settings.json` | 会话停止时只读评审闸门（默认开） |
| SessionEnd → lifecycle | `~/.codebuddy/settings.json` | 会话结束生命周期处理 |

## 9. 维护约定

- 新增规则：写入 `AGENTS.md`/`CLAUDE.md`（或全局设定），并在第 8 节标注 Tier 与 MUST/SHOULD/MAY。
- 升级为 MUST 的规则：评估是否需下沉到 `user-prompt-submit` hook 或 `sandbox`，使其不被聊天指令绕过。
- 本文件与 `AGENTS.md` 冲突时：以本文第 1–4 节的优先级裁决为准；若需长期改变优先级，先改本文。
- 每次实质性工作后，按工作流在 `.workbuddy/memory/` 记录关键决策。
