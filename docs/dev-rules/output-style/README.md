# Output Style：渐进式陪跑教学

本规则控制 Agent 的回复、维护性文档和代码注释。它解决的不是“写得更口语”这么简单，而是让聪明但不熟悉当前领域的读者能够逐步建立准确理解，同时保留工程深度。

可以把这种写法直觉地理解为“先带读者走上坡，再给地图”：先用低门槛语言建立方向感，再补上术语、机制和边界。这个类比的边界是，最终内容仍要精确、可验证，不能只停留在故事或比喻层面。

## 一、语言规则

1. 面向用户的回复默认使用中文。
2. 仓库中的维护性文档默认使用中文，包括 `AGENTS.md`、README、roadmap、exec plan、spec、ADR、research 和 skill 文档。
3. 根 `README.md` 固定为英文产品入口，`README.zh.md` 固定为中文版本；两者必须表达同一产品合同，不能把任一份改成另一份的同语言副本。
4. 专业术语、命令、代码标识符、协议字段、工具名和必要的原文名称可以保留英文；首次出现且读者可能不熟悉时，用一句中文解释。
5. 代码注释使用英文。面向用户的错误消息、CLI 输出、测试数据、fixture 和引用的外部原文不属于代码注释，可以按其实际用途选择语言。
6. 不为了“全中文”生硬翻译业内已有稳定含义的术语，例如 `roadmap`、`gate`、`fixture`、`Graphify`。应解释其在当前语境中的作用，而不是只替换字面名称。

## 二、渐进式表达规则

解释复杂概念或编写教学性、设计性、决策性文档时，按以下顺序组织：

1. **先给结论**：先让读者知道这一段要解决什么，以及最终判断是什么。
2. **再给直觉**：必要时使用一个贴近读者已有经验的类比，帮助建立方向感。
3. **再给准确说法**：补充正式术语、机制、约束和可验证事实。
4. **说明类比边界**：指出类比在哪些地方不成立，避免读者把辅助理解误当成完整模型。
5. **解释为什么**：优先说明选择背后的因果和权衡，不堆百科式定义。
6. **控制概念密度**：每个小节围绕一到两个核心点；需要更多概念时分层展开。
7. **提示关键误区**：主动指出最可能造成错误行动的误解，但不要把正文带入无关扩展。
8. **压缩主线**：复杂回答或长篇说明结尾增加简短总结，帮助读者重新抓住结论、原因和下一步。

不是每一段都必须机械套用类比。简单命令、表格字段、变更日志和明确的操作步骤应直接表达；只有存在理解门槛时才增加直觉层。渐进式表达的目标是降低认知门槛，不是增加篇幅。

## 三、交互规则

1. 用户按编号提问时，按相同编号逐条回答；每一点先写结论，再解释原因。
2. 用户复述自己的理解时，明确分成“理解正确的部分”和“需要微调的部分”，并说明为什么需要调整。
3. 语气可以口语化、陪聊式，但判断和边界必须明确，不能用轻松语气掩盖不确定性。
4. 一次优先讲清一到两个关键概念。可选扩展只做提示，等待用户需要时再展开。

## 四、文档编写规则

文档不是把聊天语气直接搬进 Markdown。文档需要稳定、可扫描，并让后来者不依赖当前对话也能理解。

- 标题和开头先说明文档用途、适用对象与结论。
- 新术语首次出现时就地解释，不把理解成本全部推给术语表。
- 设计文档和 ADR 重点写“为什么选择、放弃了什么、边界在哪里”。
- roadmap 先讲目标和非目标，再列阶段；exec plan 先讲当前主线，再列动作与验证。
- 操作指南先给最短可行路径，再补失败分支和深入原理。
- 长文结尾使用“主线总结”“决策摘要”或同等含义的小节压缩重点。
- 代码块、命令、路径、字段名和机器可读合同保持原样，不能为了语言统一破坏可执行性。

### 人类与 Agent 共用的快速读取协议

维护性长文和目录 README 必须让人类与 AI Agent 使用同一个首屏入口，不能分别维护两套摘要。首屏优先使用以下稳定字段：

| 字段 | 要回答的问题 |
|---|---|
| 文档角色 | 这份文档负责什么，不负责什么？ |
| 当前状态 | `proposed`、`active`、`completed`、`superseded` 或其他明确生命周期是什么？ |
| 当前结论/入口 | 现在真正有效的结论或必读文件是哪一个？ |
| 何时读取 | 哪类任务需要读，读到哪里可以停止？ |
| 何时更新 | 什么变化必须同步维护这份文档？ |
| 关联真源 | 发生冲突时应回到哪份 spec、ADR、roadmap、plan 或规则？ |

人类主要通过标题、表格和结论快速扫读；Agent 主要依赖稳定字段、状态值和明确链接建立上下文。两者需要的是同一组事实，所以不能用只对机器友好的 front matter 替代正文，也不能用只有叙事、没有状态和链接的长文要求 Agent 猜测。

目录 README 还必须提供“按问题选择文档”的路由，避免要求读者先读完整个目录。重复当前进度时只链接唯一真源，不复制易变化的任务状态和百分比。

## 五、适用范围与例外

本规则适用于 Agent 新增或修改的第一方维护性内容：

- 根目录的 `AGENTS.md`、`CONTEXT.md`、`README*.md`；
- `docs/` 下的活动说明、ADR、research、roadmap、exec plan、spec、aiprompts 和 dev-rules；
- `skills/repo-analyzer/` 下的 skill 与 references 文档。

以下内容不做追溯改写：

- `docs/archive/`：归档记录必须保留历史原貌；
- `tests/baseline/`：测试运行证据必须保持可核验，不能为了文风改写；
- `vendor/`：第三方参考源码不由本仓库规则重写；
- 测试 fixture、生成报告、转录、缓存和机器生成工件；
- 外部原文引用，但引用后的解释仍应遵守本规则。

## 六、提交前检查

修改回复规范或文档时，至少检查：

1. 正文是否以中文为主，保留的英文是否属于专业术语、标识符或必要原文。
2. 新术语首次出现时是否给出了读者能行动的一句话解释。
3. 复杂段落是否先给结论，并解释了原因和边界。
4. 是否出现无助于理解的术语堆叠或机械类比。
5. 长文结尾是否有主线压缩。
6. 代码注释是否为英文，代码块和机器合同是否保持可执行。

## 主线总结

回复和文档都以中文为主，代码注释使用英文。遇到复杂内容时，先给结论和直觉，再给准确机制与边界；遇到简单内容则直接说清楚。目标不是把内容写长，而是让读者在不牺牲技术深度的前提下，始终知道“现在讲什么、为什么重要、下一步是什么”。
