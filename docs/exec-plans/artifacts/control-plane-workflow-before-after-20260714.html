完整门收口流程：实施前 vs 实施后

这份对照页解释同一类交付（例如 Skill 原子交付架构迁移）在 没有防呆 与 加上 awaiting-judge + 独立 Judge + 校验脚本 之后，状态如何流动，以及为什么后者能堵住“测试绿了就标 completed”的假完成。

场景：完整门 / 必须独立 Judge 事故点：假 completed 制品日期：2026-07-14
并排流程 上次事故落点 为什么能解决 脚本何时触发 一句话公式

实施前

规则写在文档里，但 completed 没有机械刹车。Worker 可以“做完 + 自测 + 写备注”直接收口。

可跳格
用户要求实施 roadmap Agent 进入端到端交付模式
↓
读部分规则并开始实现 可能读 quality-gates，但不一定先把 dual-agent 跑成硬门
↓
Worker 连续完成 D0–D3 迁移代码、改测试、更新文档
↓
跑测试，结果变绿 32 passed 等强反馈出现
↓
错误跳跃：直接标 completed 用“实现完成 + 自测通过”替代“审查通过”
↓
可选：在 progress 写“流程缺口” 诚实披露存在，但状态已经被抬高
失效模式：披露替代门禁。系统允许“没 Judge pass 也能 completed”。

实施后

把审查做成状态，把 completed 做成校验结果。Worker 只能把活做到 awaiting-judge，不能自己盖竣工章。

不可跳格
用户要求实施 roadmap 先分类：完整门 + 独立 Judge=必须
↓
激活 plan，写启动关字段 质量门 / 独立Judge / 工作区基线；可选 bootstrap 校验
↓
Worker 实现并自验 代码、测试、Boundary Check、未执行项如实记录
↓
强制停点：awaiting-judge 产出 Judge 输入包；同会话禁止 completed
↓
独立 Judge 只读审查 至少一项独立验证；输出 Verdict: pass / revise / blocked
↓
严格校验脚本 检查 progress 是否有 pass 或用户豁免
↓
只有校验通过，才允许 completed 状态升级被机器门卡住，不再靠自觉
防呆点：测绿只证明 Worker 做完；completed 只能由 Judge pass（或豁免）+ 脚本放行解锁。

上次事故落在哪一格

不是“完全不知道要 Judge”，而是在测绿之后换了完成标准。

实现阶段
D0–D3 迁移、测试重组、旧入口删除——这些都做了。
自测阶段
unit/contract/install 通过，metadata 校验通过——强反馈让“完成感”出现。
本应停住
正确状态应是 awaiting-judge：实现暂停，等待独立审查。
实际跳跃
跳过独立 Judge，把“流程缺口”写进 progress，同时把 plan/roadmap 标成 completed。
根因一句话
用“证据披露”完成了合规叙事，却没有用“状态门禁”完成合规控制。

为什么实施后就能解决这个问题

因为它不再依赖 Agent“想起来要守规矩”，而是改变了默认可走路径。

1. 给半成品合法位置

实施前：测绿后只剩“继续标 completed”显得完整。 实施后：测绿后合法状态是 awaiting-judge。 Agent 不必靠假完成来交差。

2. 拆开两种“完成”

实现完成：Worker 自测绿。
验收完成：Judge pass 或用户豁免。
以前两者混用一个 completed；现在必须分两步。

3. 用机器门取代自觉

脚本检查：若 plan=completed，progress 必须有 Verdict: pass 或豁免段。 没有就失败。备注再诚实也不能过门。

对照表

检查点 实施前 实施后
Worker 自测通过后能做什么 可以直接写 completed 只能进入 awaiting-judge
独立 Judge 文档要求，但可跳过并写备注 状态升级前置条件
progress 写“流程缺口” 像合规，实则仍可 completed 校验仍失败，状态升不上去
completed 的含义 “我做完了” “审查放行了（或你豁免了）”
失败是否可静默 可以，靠叙述掩盖 脚本/CI/收口前命令会暴露

它解决的是哪个问题，不解决哪个

问题 是否解决
假 completed（没 Judge 却标完成） 解决。这是主目标。
Worker 把披露当免罪符 解决。披露不再解锁状态。
测试没发现的产品 bug 部分缓解。Judge 可能抓住，但不保证。
真实外部安装 / G5 UAT 是否通过 不解决。那是更高证据层，需 real-uat-regression。

校验脚本什么时候触发

主触发（硬门）

任何把 plan/roadmap 标成 completed 之前， 必须先跑严格模式校验。失败则保持原状态。

副触发（审计网）

CI / release preflight 再跑严格模式，防止有人手工改文件绕过 Agent 纪律。

早触发（可选）

完整门激活时跑 bootstrap：只查“质量门 / 独立Judge / CLOSE-J 任务”字段是否写了，不要求已有 Verdict。

不触发

日常每刀写代码、轻量门小修、只读浏览历史 completed 文档——避免噪音导致规则被关掉。

一句话公式

Worker 做完 → awaiting-judge → Judge pass / 豁免 → 脚本放行 → completed

实施前缺的是中间两格；实施后中间两格变成默认必经之路。 所以“为什么实施就解决了”：因为它修的不是说明书，而是状态升级路径。
