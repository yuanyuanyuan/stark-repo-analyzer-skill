# 限定程序化控制面为 Graphify 闸门

Graphify 增强流程必须通过 Graphify 前置闸门，但程序化控制面只负责版本检查、code-only 建图、最小健康验证、来源路径规范化和导航 map 生成。模式交互、模块划分、源码阅读、覆盖率判断、交叉验证与最终报告继续由参考 repo-analyzer 的 Agent 流程负责，因为 Graphify 只提供候选导航，不能取代分析判断，也不应把一次集成改造成独立分析应用。

## 影响

- `analyze`、`finalize`、`validate`、`resume`、完整运行 manifest 和模块草稿格式校验不再构成普通用户的产品主路径；仍有价值的确定性检查可以保留为开发期测试工具。
- 用户分析运行只强制保留 Graphify 原始图/报告、规范化图/报告、导航 map 和健康或失败摘要；额外 manifest、全目录哈希和固定草稿格式只属于开发验收运行。
- 最终用户交付保持为单一 `ANALYSIS_REPORT.md`；Graphify 产物和草稿只是工作区支撑证据，不扩张成 HTML 仪表盘或多份正式报告。
- 最终报告只简述 Graphify 如何影响阅读路径、哪些候选被源码确认或否决，以及是否使用原版兼容流程；节点、边、doctor 和完整图谱细节留在工作区证据中。
- Graphify 增强流程开始建图后，闸门失败必须终止，不能静默降级；仅依赖不可用且用户暂不安装时可以在明确标记后进入原版兼容流程。
- `standard` 与 `deep` 共享同一 Graphify 闸门，分析档位差异留在 Agent 的源码阅读与验证阶段。
