# 重实现验收清单

后续 `stark-repo-analyzer-skill` 重实现完成后，使用本清单与参考输出基线逐项比较。

## 输入与范围

- [ ] 能识别仓库路径、`owner/repo`、GitHub/GitLab/Gitee URL 和项目名称输入。
- [ ] 能明确区分当前项目、参考语料、目标分析项目和输出工作目录。
- [ ] 本地源码输入不依赖重新 clone，且能记录固定 commit。
- [ ] 不使用目标项目 Git 历史推断当前实现，除非用户明确要求历史演进分析。

## 分析模式与规模

- [ ] 能统计有效代码规模，并排除测试、生成物、锁文件和文档代码。
- [ ] 能解释 `快速/standard/deep` 的覆盖率目标和适用条件。
- [ ] `standard` 默认目标为核心模块至少 60%、次要模块至少 30%。
- [ ] 大型仓库可以进入 bounded scope，但必须记录分母、已读范围和未读原因。
- [ ] 不把源码阅读覆盖率称为测试覆盖率或运行时覆盖率。

## 研究与提问

- [ ] 能根据项目特征生成问题，而不是套用固定问题列表。
- [ ] 能区分代码事实与外部调研事实，并保存外部来源。
- [ ] 研究至少覆盖项目问题、定位、同类路线和组织动机；找不到时明确写“未找到”。
- [ ] 能在报告详略、场景引入和竞品定位方面与用户对齐。

## 模块分析

- [ ] 按业务能力、数据流和职责划分模块，而不是按目录机械分章。
- [ ] 每个核心模块说明整体角色、业务问题、设计思路、核心数据结构、执行流程和模块依赖。
- [ ] 核心流程包含 Mermaid 图，并引用源码路径和行号。
- [ ] 每个设计判断说明 Why、替代方案代价、边界和对整体设计哲学的影响。
- [ ] 模块叙事线能解释上一个模块如何引出下一个模块。

## 交叉验证与报告

- [ ] 每个项目生成单一 `ANALYSIS_REPORT.md`。
- [ ] 报告包含问题/定位、项目全景、深度模块分析、评价与启发、架构图和限制。
- [ ] 能从草稿中抽查关键结论并回到源码验证。
- [ ] 跨模块结论有独立交叉验证记录。
- [ ] 不确定、未读、未运行和外部依赖均明确标记。

## 产物与复现

- [ ] 每个项目保存 `metadata.json`、`execution-log.md`、`checks.md` 和覆盖率汇总。
- [ ] metadata 记录源码路径、commit、分析模式、工具、模型、开始/结束时间和运行结果。
- [ ] 输出目录彼此隔离，单文件不超过参考 skill 的分块限制。
- [ ] 失败、超时、外部工具不可用和运行时副作用均写入日志。
- [ ] 可用同一个固定 commit 和输入提示重新执行，并比较输出差异。
- [ ] 并行 agent 的数量、职责、完成状态和接管情况有独立编排记录。

## 代表性样本回归

- [ ] `click`：命令树、Context、参数和终端体验仍能被识别。
- [ ] `httpx`：消息模型、流生命周期、Transport 边界和配置路由仍能被识别。
- [ ] `ruff`：CLI、配置、lint、formatter 与大型未覆盖模块边界仍能被区分。
- [ ] `codex-plugin-cc`：Job、Broker、App Server 和宿主 hooks 的生命周期仍能被识别。
- [ ] `claude-code`：Query loop、Tool runtime、Task/agent、Context/memory 和扩展边界仍能被识别。
- [ ] `codex`：Session/turn、Protocol/Event、Execution safety、TUI 和扩展注入主线仍能被识别。
