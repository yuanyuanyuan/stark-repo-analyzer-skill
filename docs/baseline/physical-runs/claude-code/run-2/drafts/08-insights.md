# 洞察草稿

## 系统性设计哲学

源码反复选择显式状态机、边界适配器和后台预取：启动阶段优化等待，工具阶段控制副作用，MCP 阶段隔离外部能力，传输阶段吸收网络抖动。这不是单纯的 CLI，而是一个把模型当作决策源、把本地进程当作受控执行器的 runtime。

## 亮点

- 启动副作用提前并行，直接针对 CLI 冷启动这一用户可感知指标。
- 传输层区分可重试断线与永久拒绝，并保留消息缓冲，避免把网络问题升级为会话丢失。
- MCP 的配置、连接、权限和资源发现分层，便于企业策略介入。

## 风险与重设计建议

- 入口导入面过宽，建议进一步拆分 composition root 与运行模式装配器。
- 多传输协议由环境变量选择，建议把能力协商和策略对象化，减少隐式分支。
- 全库规模极大，建议建立模块级架构测试，验证工具批准、会话恢复和远程重连之间的契约。
