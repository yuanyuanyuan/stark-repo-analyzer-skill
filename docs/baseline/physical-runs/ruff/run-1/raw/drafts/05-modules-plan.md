# 阶段 5：模块与叙事计划

## 全局设计判断（待模块验证）

Ruff 的组织方式不是把“快”当成孤立优化，而是把兼容性、配置和多种 Python
开发动作收束为单一运行面：CLI/设置层决定解释规则，lint 与 formatter 则共享
解析相关基础设施、各自保有语义边界。该判断将在阶段 7 用模块证据交叉验证。

## 叙事线

```mermaid
flowchart LR
    U[用户命令与路径] --> C[CLI 与配置编排]
    C --> L[Lint 分析与安全修复]
    C --> F[Python 格式化管线]
    L --> O[诊断/修改结果]
    F --> O
```

1. **CLI 与配置编排**：先解释统一命令如何把路径、CLI 覆盖项和最近配置转为
   每个文件可执行的设置；它的输出是 lint/format 路径的输入契约。
2. **Lint 分析与修复**：设置已确定后，解释规则选择、诊断生产和 safety-aware fix
   如何将“兼容广度”变成可控自动化；其结果再交回 CLI 呈现。
3. **Python 格式化管线**：同样的输入契约进入格式化器；重点是 AST、注释与文档
   IR 如何保证一致输出和 Black 迁移预期，而不与 lint 语义混淆。

## 模块分级

| 模块 | 类型 | 前置问题 | 为下一节铺垫 |
|---|---|---|---|
| CLI 与配置编排 | 核心 | “用户意图和目录差异如何被解释？” | 确定 lint/format 的设置与输入 |
| Lint 分析与修复 | 核心 | “规则如何变成安全、可呈现的诊断？” | 解释统一 CLI 的检查半边 |
| Python 格式化管线 | 核心 | “格式化如何保证兼容和源文本保真？” | 与 lint 一起完成统一工具链图景 |

次要模块没有独立 agent：本次 bounded scope 下，parser/AST、缓存、notebook、
server、WASM、ty 和规则实现均在范围外，避免用浅阅读伪造 30% 次要覆盖。
