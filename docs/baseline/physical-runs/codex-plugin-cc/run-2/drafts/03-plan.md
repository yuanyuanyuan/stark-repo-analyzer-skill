# 阶段 3：规模与分析计划

## 固定选择

- 分析模式：`standard`（用户指定）。
- 核心模块最低覆盖率：60%；次要模块最低覆盖率：30%。
- 未执行交互式模式选择和报告详略询问：模式与输出合同已由调用方固定。

## 规模统计

脚本实现文件共 5,349 行：`plugins/codex/scripts/**/*.mjs`，统计排除了测试、`package-lock.json`、Markdown、JSON、许可证与构建配置。`wc -l` 输出的合并尾行本身会重复累计，已按各个脚本文件行数求和，未采用重复的 `total` 行。

测试文件为 3,656 行，作为行为证据阅读，不计入有效实现行数。最大实现文件为 `scripts/lib/codex.mjs`（1,219 行）及 `scripts/codex-companion.mjs`（1,073 行）；两者需要分段阅读。

## 分析问题

1. 插件如何把面向 Claude 的命令变为面向 Codex app server 的同步/后台调用？
2. 后台 job 的身份、结果和取消如何跨进程与跨 Claude 会话保持可追踪？
3. 可选 Stop 审查门控如何在不破坏正常结束语义的前提下阻止有问题的输出？

## 验证计划

- 逐模块收集带行号的实现与测试证据，并在覆盖率表中记录实际读取范围。
- 静态语法检查只使用 `node --check`，不运行会写入临时目录或修改源树的构建/测试命令。
- 最终检查输出边界、源 HEAD、各草稿存在性、Markdown Mermaid 围栏与覆盖率门控。
