## Problem Statement

Repo Analyzer 2.0.0 已经通过确定性 Repo Map、关键单元分母、Evidence Matrix 和质量门建立了可审计的 Evidence-First 分析流程，但在大型、陌生或语义边界复杂的公共仓库中，主 Agent 仍需要花费大量时间判断“应该先看什么、可能遗漏什么、哪些设计张力值得追问”。当前确定性扫描擅长发现文件、符号、入口和引用候选，却不负责形成业务能力、概念关系、设计动机或跨仓库技术路线的语义地图。

DeepWiki MCP 可以在数秒到数十秒内返回公共 GitHub 仓库的主题结构、仓库问答和多仓库比较结果，有机会减少早期盲读并提高模块、跨模块关系和风险路径的候选召回率。但实测表明，DeepWiki 不暴露可信的索引 commit、branch 或时间，不能稳定提供 file:line 引用，可能返回陈旧路径或无法验证的自然语言结论；未索引仓库还会以 HTTP 成功和业务错误文本的形式返回失败。

用户需要在 v2.1.0 中安全地获得 DeepWiki 的语义发现能力，同时保持当前 commit 的本地源码为行为事实来源，避免外部生成内容污染 coverage、Evidence Matrix、质量门和最终报告。

## Solution

v2.1.0 引入可选的 DeepWiki Scout，将 DeepWiki 定位为 Evidence Discovery Accelerator，而不是证据源或替代分析引擎。

工作流先基于确定性 Repo Map、关键单元、项目文档和用户目标独立生成 P0 Evidence Plan；随后 DeepWiki Scout 以结构查询和少量定向问答执行 gap challenge，提出 P0 可能遗漏的业务模块、跨模块关系、风险路径、设计张力和竞品比较问题。所有 DeepWiki 输出统一归类为 external hypothesis。只有在当前 checkout 上重新读取源码、验证依赖两端并获得有效 file:line 锚点后，候选才能进入 P1 Evidence Plan、Evidence Matrix、coverage 或最终确定性叙事。

DeepWiki Scout 默认以 shadow mode 交付并记录增量贡献、延迟、上下文成本和验证结果。quick 模式默认不调用；standard 模式使用一次结构查询和少量定向问题；deep 模式可以增加风险、替代方案和多仓库比较。未索引、超时、服务不可用、本地仓库、私有仓库、脏工作区或索引 commit 无法确认时，基线 Evidence-First 流程必须继续完成且不降低质量要求。

Python CLI 作为 DeepWiki MCP 的默认命令行适配实现，但 v2.1.0 的领域契约面向 provider 能力和标准化研究工件，不绑定具体传输实现。

## User Stories

1. As a repo-analyzer 使用者, I want 大型公共仓库快速获得语义主题地图, so that 我不必从目录和 README 开始盲目探索。
2. As a repo-analyzer 使用者, I want DeepWiki 只作为可选增强, so that 服务不可用时仍能完成完整分析。
3. As a repo-analyzer 使用者, I want 分析继续以当前 checkout 为事实来源, so that 远端索引不会覆盖本地代码现实。
4. As a repo-analyzer 使用者, I want 未提交代码和脏工作区默认不使用 DeepWiki 判断当前行为, so that 陈旧远端内容不会误导报告。
5. As a repo-analyzer 使用者, I want quick 模式默认跳过 DeepWiki, so that 小型和限时分析不会承担网络与上下文开销。
6. As a repo-analyzer 使用者, I want standard 模式获得有限且高价值的语义查漏, so that 常规分析在成本和洞察之间保持最佳平衡。
7. As a repo-analyzer 使用者, I want deep 模式能够比较多个仓库的设计路线, so that 竞品分析聚焦架构权衡而不是功能清单。
8. As a repo-analyzer 使用者, I want DeepWiki 的调用耗时和输出规模被记录, so that 我能判断它是否真正节省总成本。
9. As a repo-analyzer 使用者, I want 完整 Wiki 内容默认不进入模型上下文, so that 大响应不会耗尽 token 预算。
10. As a repo-analyzer 使用者, I want 未索引仓库快速降级, so that 一次不可用的外部查询不会阻塞整个分析。
11. As a 主 Agent, I want 在查看 DeepWiki 前先生成独立 P0 Evidence Plan, so that 外部叙事不会提前锚定模块分类和报告故事线。
12. As a 主 Agent, I want 比较 P0 与 DeepWiki 主题地图的差异, so that 我能区分本地候选、外部新增候选和双方冲突。
13. As a 主 Agent, I want DeepWiki 新增候选明确标记来源和验证状态, so that 假设不会静默升级为事实。
14. As a 主 Agent, I want DeepWiki 定向提出错误处理、缓存、并发、安全和兼容风险, so that 风险抽样不只覆盖 happy path。
15. As a 主 Agent, I want DeepWiki 发现的跨模块关系重新验证依赖两端, so that 概念关联不会被误写成真实调用关系。
16. As a 主 Agent, I want 只有本地验证后的候选才能进入 P1 Evidence Plan, so that DeepWiki 的增量价值可审计。
17. As a subagent, I want 明确知道 external hypothesis 不能作为源码证据, so that 模块草稿不会引用生成式综述代替代码。
18. As a subagent, I want 获得经过筛选的候选模块和风险问题, so that 有限证据预算优先用于高价值源码路径。
19. As a repo-analyzer 维护者, I want 一个稳定的 external research provider 接缝, so that DeepWiki 的传输实现可以替换而不改动分析语义。
20. As a repo-analyzer 维护者, I want DeepWiki 响应被标准化为版本化研究工件, so that缓存、审计、测试和后续 provider 能复用同一契约。
21. As a repo-analyzer 维护者, I want 识别 HTTP 成功但正文表示未索引的业务错误, so that 不可用响应不会被误判为有效研究。
22. As a repo-analyzer 维护者, I want 为 DeepWiki 设置调用次数、超时和总预算, so that 外部研究不会吞噬 quick、standard 或 deep 的整体预算。
23. As a repo-analyzer 维护者, I want provider 故障触发 circuit breaker, so that 第一次失败后不会继续产生无意义调用。
24. As a repo-analyzer 维护者, I want 缓存记录本地 commit、获取时间和索引 commit unknown 状态, so that 重复分析不会伪装成新鲜证据。
25. As a repo-analyzer 维护者, I want gate 验证锚点对应真实文件和有效行范围, so that 看似正确的 file:line 文本不能通过机械检查。
26. As a repo-analyzer 维护者, I want gate 拒绝 external hypothesis 直接进入 source evidence 或 coverage, so that DeepWiki 永远不能自行证明架构结论。
27. As a repo-analyzer 维护者, I want 默认测试使用 fake provider, so that测试稳定、快速且不依赖 DeepWiki 网络服务。
28. As a repo-analyzer 维护者, I want 可选 live probe 测试验证协议兼容性, so that服务契约变化能够被发现但不影响普通 CI。
29. As a release 负责人, I want A/B benchmark 比较无 DeepWiki 与 shadow mode, so that v2.1.0 的效率和质量收益有证据而不是假设。
30. As a release 负责人, I want 质量安全门优先于速度收益, so that token 或时间下降不会以错误结论增加为代价。
31. As a security reviewer, I want 本地、私有和脏工作区默认不发送问题或源码上下文, so that 未公开信息不会泄露到第三方服务。
32. As a security reviewer, I want DeepWiki 请求只包含公开仓库标识和经过约束的问题, so that外发数据保持最小化。
33. As a 后续 provider 实现者, I want 复用相同 artifact 和验证状态模型, so that接入其他语义服务不需要重新设计 Evidence-First 边界。
34. As a v2.1.0 实现 Agent, I want 每个交付切片可独立演示和验证, so that多个 Agent 可以按依赖关系安全推进。

## Implementation Decisions

- DeepWiki 的领域角色固定为 Evidence Discovery Accelerator，所有原始输出和派生判断默认属于 external hypothesis。
- 传输层默认调用现有 Python CLI 中转 DeepWiki MCP；分析工作流依赖 provider 接口，不依赖 Python 包内部实现。
- 新增一个最高层 provider-to-artifact 接缝，负责资格判断、调用预算、超时、业务错误识别、标准化输出、缓存元数据和可观察指标。
- 研究工件至少记录 schema 版本、provider 与服务版本、仓库标识、本地 commit、获取时间、分析模式、问题、原始响应、可用状态、调用延迟、响应规模、索引 commit 状态、验证状态和错误原因。
- 索引 commit、branch 或时间无法从结构化协议确认时必须写 unknown；不得通过自然语言追问生成这些元数据。
- 工作流顺序固定为 P0 Evidence Plan、DeepWiki gap challenge、本地源码验证、P1 Evidence Plan。P0 必须在 DeepWiki 内容进入模型上下文前形成并保留。
- gap challenge 输出区分 baseline-only、DeepWiki-only、agreement 和 conflict。agreement 不构成独立双源验证，因为两者仍可能解释同一份源码。
- DeepWiki-only 和 conflict 项必须进入待验证候选或开放问题；只有当前 checkout 的源码、项目一手文档或可验证外部一手来源可以把它们升级为确定性判断。
- quick 模式默认禁用 DeepWiki；仅在用户显式启用、仓库已索引且本地规模信号表明有价值时允许一次结构查询。
- standard 模式预算为一次结构查询和最多两个定向问题，优先覆盖核心数据流、模块边界和风险路径。
- deep 模式可以增加定向风险问题、替代方案问题和多仓库比较，但必须受总调用次数、总耗时和上下文预算约束。
- 默认禁止调用完整 Wiki contents。只有用户明确请求读取完整 DeepWiki 文档且剩余预算允许时才可执行，并需先显示规模风险。
- 首次结构查询同时承担索引可用性探测。未索引、超时、协议错误、限流或服务不可用时立即停止后续 DeepWiki 调用并回到基线流程。
- HTTP 状态和 MCP isError 字段不能单独代表业务成功；适配器必须识别正文中的未索引和服务错误结果。
- 本地无公共 GitHub 远端、私有仓库、脏工作区或当前 commit 与远端默认分支不一致时，DeepWiki 不得用于当前行为判断。深度分析可在明确标记后把它作为远端历史背景，但不能进入正式证据链。
- DeepWiki 内容不得直接写入关键单元 analyzed 状态、coverage 分子、Evidence Matrix source evidence、风险 evidence 或确定性报告结论。
- 质量门增加证据来源边界检查，并验证 file:line 锚点对应当前仓库中存在的文件和有效行范围。语义蕴含仍由模块交叉验证或后续专门质量机制承担。
- DeepWiki 研究工件和预算执行情况进入最终运行摘要，包括调用次数、累计延迟、响应规模、新增候选数、本地验证成功数、无效或陈旧候选数和降级原因。
- shadow mode 是 v2.1.0 初始默认策略。它允许记录 DeepWiki 的候选和验证结果，但是否影响正式 P1 和最终报告由发布门槛决定。
- A/B benchmark 固定仓库 commit、模型、模式、总预算和并行度，比较 baseline 与 DeepWiki shadow 两条流程。
- v2.1.0 的长期启用门槛为：Evidence Plan 时间显著下降、本地源码读取量下降、经验证的新关系或风险发现增加，同时 gate 首次通过率、错误确定性结论、无效锚点和 Unsupported Claims 不得恶化。
- 当前 package 版本保持 2.0.0；实现和发布流程负责在发布时升级到 2.1.0，本规格不直接修改版本文件。

## Testing Decisions

- 主要测试接缝是从 repo-analyzer 命令输入到标准化研究工件、Evidence Plan 差异和 gate 结果的外部可观察行为。内部 HTTP、SSE 或 Python 实现不作为主要断言对象。
- 默认测试使用可注入 fake provider，固定模拟有效结构、有效问答、未索引、超时、协议错误、限流、陈旧路径和无法验证的 commit 信息。
- provider 测试验证相同输入产生稳定 artifact，错误状态不会被写成有效研究，并且失败后不会继续调用。
- 工作流测试验证 P0 在 DeepWiki 之前生成、P1 保留 P0 来源、所有新增项拥有 provider provenance 和本地验证状态。
- 模式测试验证 quick 默认零调用、standard 调用预算受限、deep 才允许多仓库比较，且完整 contents 默认被拒绝。
- 资格测试覆盖公开且干净的 GitHub checkout、本地无远端、私有远端、脏工作区、分支不一致和未索引仓库。
- gate 测试验证不存在的文件、越界行号、external hypothesis 锚点和仅有 URL 的证据不能进入 coverage 或 Matrix 的正式证据字段。
- 降级测试验证 DeepWiki 不可用时 doctor、scan、summarize、units、Evidence Plan、模块分析和 gate 仍能按原质量契约完成。
- 安全测试验证默认不会发送本地源码、diff、未提交内容、私有仓库标识或用户未批准的敏感问题。
- live probe 作为手动或定时兼容性测试，不进入普通单元测试；它只验证 MCP initialize、tools/list、structure 和 ask 的当前协议形态。
- A/B benchmark 至少包含小、中、大公共仓库，并覆盖文档丰富/稀缺、单体/monorepo、静态/动态扩展机制；故障场景不发送真实私有代码。
- benchmark 记录 Evidence Plan 墙钟时间、DeepWiki 延迟、响应规模、近似 token、工具调用数、本地读取文件和 span、候选验证率、核心模块 recall、关系与风险发现、无效锚点、Unsupported Claims 和盲审质量。
- 进入长期可选集成至少要求 Evidence Plan 时间下降 25%、本地源码读取量下降 15% 或经验证的新关系/风险发现增加 10% 中的实质收益，同时错误确定性结论、无效锚点和 gate 失败率不得增加。
- 现有 CLI fixture 与工件断言风格是 prior art；测试优先验证产物契约和退出码，不绑定内部命令拼接细节。

## Out of Scope

- 不让 DeepWiki 替代 Doctor、Repo Map、关键单元枚举、Graphify、Evidence Matrix 或 gate。
- 不把 DeepWiki 文本当作当前源码行为、作者动机或跨模块调用的最终证据。
- 不自动访问 DeepWiki 页面为未索引仓库触发索引。
- 不在 v2.1.0 接入 DeepWiki 私有模式、认证仓库或 Devin 私有工具。
- 不默认发送本地源码、git diff、未提交内容或私有仓库信息。
- 不默认读取或注入完整 Wiki contents。
- 不构建本地向量数据库、embedding 服务或新的重量级语义索引器。
- 不承诺 DeepWiki 的可用性、定价、索引新鲜度或第三方 SLA。
- 不把多仓库 DeepWiki 比较替代每个项目各自的完整 Evidence-First 分析。
- 不在本规格中修改 package 版本、发布标签或 changelog；这些属于实现完成后的发布流程。

## Further Notes

- 2026-07-10 实测的 DeepWiki 公共 MCP 服务版本为 2.14.3，公开工具包括 wiki structure、wiki contents 和 repository question answering；问答支持最多十个仓库。
- 实测已索引大型仓库的 structure 通常在数秒内返回，定向架构问答约十几到二十多秒。完整 Ruff Wiki 的原始 MCP 响应约 1.89 MB，证明 contents 不适合作为默认上下文输入。
- DeepWiki 对 Ruff 的架构问答成功发现多个真实核心入口和模块，但未提供 file:line，并给出一个已变化的源码路径；另有测试生成无法在 GitHub 验证的 commit SHA。因此它适合提高候选召回，不适合提高证据确定性。
- 当前仓库和 deepwiki-cli 在实测时均未被 DeepWiki 索引，所以 v2.1.0 的 benchmark 必须选择已索引仓库并单独覆盖未索引降级。
- 仓库当前声明版本为 2.0.0。根据维护者决定，本需求属于下一向后兼容功能版本 2.1.0。
